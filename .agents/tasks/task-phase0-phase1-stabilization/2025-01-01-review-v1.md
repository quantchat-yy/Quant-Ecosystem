# Monorepo build and type safety stabilization

This change makes a 57-package TypeScript monorepo pass `pnpm typecheck`, `pnpm build`, and `pnpm test` by fixing the Turborepo task graph, adding missing `build` scripts to 16 packages, integrating Prisma generation into the typecheck pipeline, and resolving ~900 TypeScript errors across 14+ packages. The TypeScript fixes are overwhelmingly `noUncheckedIndexedAccess` non-null assertions, unused-variable cleanup, and Prettier reformatting. Six page-level components were deleted to eliminate unfixable type errors in Next.js apps.

Watch for: Non-null assertions (`!`) on array accesses are used pervasively without preceding bounds checks (confirmed). A committed `tsconfig.tsbuildinfo` artifact will create merge noise (confirmed). The `useAnalytics` hook was duplicated across apps with a stale "QuantAds" comment header (confirmed, low severity).

## High-level view

The turbo.json change (`typecheck` depends on `^build` instead of `^typecheck`) is the structural fix that makes everything else work: packages need their dependencies built (emitting declarations) before they can typecheck. Adding `"build": "tsc"` to 16 library packages completes this dependency chain.

TypeScript errors were fixed through three mechanical patterns applied at scale: (1) appending `!` to array/map indexed accesses to satisfy `noUncheckedIndexedAccess`, (2) prefixing unused parameters with `_`, and (3) removing unused imports and local variables. Six Next.js pages with deep type incompatibilities were deleted rather than fixed.

The `useAnalytics` hook from `quantads` was copied verbatim into `quantedits`, `quantmax`, and `quantneon` — including the "QuantAds" header comment — rather than being extracted to a shared package.

The `search-indexer` adds test glob exclusions to its main `tsconfig.json`, which means `pnpm typecheck` for that service skips test files entirely — a gap that could hide type errors in tests.

## Non-null assertion pattern for `noUncheckedIndexedAccess`

The dominant fix pattern is:

```typescript
// Before (TS error: possibly undefined)
const item = this.queue[i].priority;

// After
const item = this.queue[i]!.priority;
```

This appears hundreds of times across observability, search, ml-pipeline, media, recommendations, and security packages. The assertion is safe when a preceding `for` loop bounds the index to `array.length`, but this invariant is structural rather than explicit — there's no `if (i < arr.length)` guard adjacent to the assertion. In code paths doing numerical computation (audio sample processing in `audio-processor.ts`, statistical tests in `canary-analyzer.ts`, feature vector math in `diversity-injector.ts`), an off-by-one in a future refactor could turn these into silent `undefined` math that produces NaN rather than crashing.

A few places use safer alternatives: `this.reports[this.reports.length - 1]!` after a `length > 0` check, or `?? 'default'` fallbacks. These are preferable but rare.

## Deleted pages as escape hatch

Six components were deleted:

- `apps/quantai/src/pages/index.tsx` (459 lines, AI chat interface)
- `apps/quantai/src/pages/voice.tsx` (341 lines, voice interface)
- `apps/quantchat/src/pages/chat/[id].tsx` (284 lines, chat conversation view)
- `apps/quantchat/src/pages/index.tsx` (184 lines, chat list)
- `apps/quantmail/src/pages/compose.tsx` (491 lines, email compose)
- `apps/quantmail/src/pages/index.tsx` (105 lines, mail inbox)

The committed `tsconfig.tsbuildinfo` for `quantneon` reveals the nature of unfixed errors in sibling apps: `noUncheckedIndexedAccess` errors in JSX components where array-derived state (`currentReel`, `currentShort`) is possibly undefined, plus `Property 'marquee' does not exist on JSX.IntrinsicElements`. The deleted pages had similar issues that would require component-level refactoring.

These pages lived in `src/pages/` of Next.js App Router projects that also have an `app/` directory. If the apps use App Router exclusively, the `src/pages/` files were dead code. If they were using both routers, deleting them breaks routes. The `app/loading.tsx` modifications with `'use client'` suggest active App Router usage, making deletions likely safe — but this warrants verification.

## Build configuration for services with tests

`services/ci-runner` and `services/git-server` got new `tsconfig.build.json` files that exclude test globs from compilation output. The `search-indexer` instead adds test exclusions to its main `tsconfig.json` — meaning `tsc --noEmit` also skips tests, so type errors introduced in test files won't surface during `pnpm typecheck`. The other two services avoid this by keeping the base config inclusive and only excluding tests for the build target.

## Committed build artifact

`apps/quantneon/dist/tsconfig.tsbuildinfo` is a large JSON build artifact tracked in git. It changes on every build and will create merge conflicts. Should be added to `.gitignore`.

## Search router silent fallback

In `services/search-indexer/src/api/search.router.ts`, the `SCOPE_INDEX_MAP[scope]` lookup now returns `Promise.resolve([])` for unknown scopes. This means a typo in a scope parameter silently returns no results rather than throwing. Whether this is fail-safe or fail-silent depends on whether the caller distinguishes "no results" from "invalid query" — worth considering if scope validation is expected upstream.
