# Brand identity system for the Quant ecosystem

A complete design token and visual identity package: 16-app registry, 6-theme system, WCAG contrast utilities, 105-icon SVG library, platform icon set generation, per-app logos, and brand guidelines documentation. The implementation is well-structured TypeScript with `as const` assertions, proper barrel exports, and a 30-test suite that validates the critical paths. The code passes typecheck, lint, and all tests.

**Watch for:** The bharat theme's primary color fails WCAG AA against its own background (3.96:1, confirmed). Maskable icon generation uses `fill="currentBackground"` which is not a valid SVG fill value (confirmed). White text rendered on most app brand colors in logo SVGs fails WCAG AA (confirmed for 14 of 16 apps). The test suite validates foreground/background contrast per theme but does not exercise primary-on-background pairs, so the bharat gap passes CI silently.

## High-level view

The theme system defines six well-differentiated palettes with a clean `Theme` interface, but the bharat theme's primary (`#D94F00`) on its warm background (`#FFF9F0`) produces only a 3.96:1 ratio, below the 4.5:1 AA threshold the guidelines promise. Since the tests only check `foreground` vs `background`, this slips through.

The icon library delivers 105 stroke-based SVGs in a consistent Feather-icon style from a single `coreIcons` object. The `generateAppIconSet` utility produces platform-size wrappers but introduces an invalid SVG fill value (`currentBackground`) in the maskable variant's background circle, which will render transparent or error in most rasterizers.

The per-app logos use white text (`fill="white"`) on each app's brand color circle. For most app colors (blues, greens, oranges, magentas) this yields contrast ratios between 2.1:1 and 4.2:1 — well below AA. These are decorative brand marks rather than informational text, so the impact depends on whether the text is relied upon for identification or is purely ornamental.

Typography gains five Indic script font stacks alongside the existing `indic` key. The addition is additive and backward-compatible, though the `devanagari` key is identical to the existing `indic` key, creating a minor redundancy.

The `generateThemeCSS` function fails silently for unknown theme names (returns a CSS comment). This asymmetry with `generateAppIconSet` (which throws) means consumers must handle two different error patterns from the same package.

## Bharat theme primary contrast gap

The bharat theme declares `primary: '#D94F00'` and `primaryForeground: '#FFFFFF'`. Against the theme's own background (`#FFF9F0`), the primary achieves only a 3.96:1 contrast ratio. Against its declared `primaryForeground` (white), it achieves 4.14:1 — still below the 4.5:1 AA threshold. This means any primary-colored text or any button with white text on the primary will not meet the accessibility standard the brand guidelines commit to.

The fix is straightforward: darken the primary slightly (e.g. `#C24500` gives ~4.7:1 against white) or shift the background warmer/lighter so the contrast gap closes. The accent color (`#7B2D8B`) passes comfortably at 7.7:1, so this is isolated to the primary slot.

The test suite checks `meetsAA(theme.foreground, theme.background)` for all themes, which passes because the bharat foreground (`#1A0F00`) against its background is 18:1. But no test validates primary-on-background or primary-on-primaryForeground, so the gap is invisible to CI.

## Maskable icon `fill="currentBackground"`

In `app-icons.ts`, the `wrapIconAtSize` function produces a background circle for maskable icons:

```typescript
const bgCircle = maskable
  ? `<circle cx="${size / 2}" cy="${size / 2}" r="${size * 0.4}" fill="currentBackground"/>`
  : '';
```

`currentBackground` is not a recognized SVG paint value. SVG supports `currentColor`, named colors, hex, `rgb()`, `url()`, and a few keywords — but not `currentBackground`. In practice this circle will render with no fill (transparent) or trigger a parse error in strict SVG rasterizers. Since maskable icons are rasterized to PNG at build time for PWA manifests, this means the safe-zone background circle will be missing.

The fix: pass the app's brand color or a solid white/dark background. Since the icon already places colored content inside, a white or theme-appropriate solid fill would be appropriate.

## White-on-brand-color contrast in app logos

Every app logo in `app-logos.ts` places a white `"Q"` (or `"M"` for Marketing) on a colored circle, then renders the app name in either `#0F172A` (light variant) or `#F8FAFC` (dark variant) beside it. The wordmark text contrast is excellent (17–18:1 in both variants). But the initial letter inside the circle fails AA for nearly every app:

- QuantMax (`#F59E0B`): 2.15:1
- QuantMaps (`#22C55E`): 2.28:1
- QuantChat (`#10B981`): 2.54:1
- QuantCalendar (`#F97316`): 2.80:1
- QuantMail (`#3B82F6`): 3.68:1

If the circle initial is purely decorative branding (the wordmark beside it provides the actual identification), this is acceptable under WCAG SC 1.4.3 which exempts logos and logotypes. But the brand guidelines document does not carve out this exception, and the `appLogos` API doesn't signal that the circle initial is decorative. Consumers who rely on the logo SVG as the sole identifier for an app (e.g., in an icon-only app switcher without the wordmark) would have an accessibility problem.

## Test coverage shape

The 30 tests cover the important integration points: registry counts, theme key completeness, contrast utility math, icon SVG validity, icon set generation, logo presence for all apps, and theme CSS output. What's missing:

- **Not tested:** Primary/background and primary/primaryForeground contrast for all themes (this is where the bharat gap hides)
- **Not tested:** The `generateFaviconSvg` function, despite being exported
- **Not tested:** `relativeLuminance` directly (only tested indirectly through `contrastRatio`)
- **Not tested:** Edge cases in `hexToRgb` (3-char hex shorthand, lowercase vs uppercase mixing, invalid input)

The `hexToRgb` function assumes 6-character hex with a `#` prefix. Passing `#FFF` or `rgb(...)` would produce `NaN` values that propagate silently through the luminance calculation, returning misleading results rather than throwing.

## Inconsistent error signaling

`generateAppIconSet('nonexistent')` throws an `Error`. `generateThemeCSS('unknown')` returns `/* Unknown theme: unknown */`. `generateAppCSS('nonexistent')` also returns a comment. The test suite validates both patterns, so this is deliberate, but it creates a split personality in the API: icon-related functions throw, CSS-generating functions fail silently. A consumer wrapping these in a build pipeline gets a broken build from one and silently broken output from the other.

## Typography redundancy

The new `devanagari` key in `fontFamily` is identical to the pre-existing `indic` key. Both resolve to `'Noto Sans Devanagari', 'Inter', system-ui, sans-serif`. The `indic` key has a JSDoc comment marking it as a "Devanagari support" fallback, so the overlap is clearly intentional (backward compat), but the duplication means consumers might use either without knowing they're the same. A JSDoc alias note on `devanagari` pointing to `indic` (or vice versa) would clarify the relationship.
