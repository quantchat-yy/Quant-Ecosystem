# Brand identity system for the Quant ecosystem (v2)

The six issues raised in the v1 review have all been addressed: bharat primary darkened to `#C24500` (now 5.07:1 vs white, 4.84:1 vs background), maskable icon uses `brandColor ?? 'white'` instead of invalid `currentBackground`, logo text switched to dark fill `#1A0F00` for light brand colors and `white` for dark ones, a test validates primary/primaryForeground AA for all themes, `generateThemeCSS` throws for unknown themes, and the devanagari/indic JSDoc documents the redundancy. All 31 tests pass, typecheck is clean.

**Watch for:** Two logo initials still fail AA despite the fill-switching fix — QuantDrive (`#6366F1`) and QuantAI (`#8B5CF6`) are mid-luminance purples where neither dark nor white text achieves 4.5:1 (confirmed). Three theme accent/destructive pairs fail AA against their declared foregrounds but are not tested: `light.accent` 3.19:1, `colorblindSafe.accent` 2.87:1, `dark.destructive` 3.76:1 (confirmed).

## High-level view

The bharat primary fix is correct and the new contrast test catches it. The maskable icon fix properly passes `app.color` through the function chain. The logo text fill-switching strategy (dark for light backgrounds, white for dark) handles 14 of 16 apps correctly, but two mid-luminance purples sit in a dead zone where neither option passes AA — these would need a lighter tint of the brand color or a different approach entirely.

The theme system has an untested gap: while `primary`/`primaryForeground` now has a test, the `accent`/`accentForeground` and `destructive`/`destructiveForeground` pairs are not validated. Two themes and one dark-theme destructive slot fail AA against their declared foreground colors. The `colorblindSafe` theme — designed specifically for accessibility — failing on accent contrast is particularly concerning.

The `generateAppCSS` function still returns a CSS comment for unknown input, while `generateThemeCSS` and `generateAppIconSet` throw. This is legacy behavior but creates a split API contract within the same module.

## Logo initial contrast dead zone

The v1 fix correctly switches to dark fill (`#1A0F00`) for light brand colors and white for dark ones (QuantEdits, Marketing). This pushes 14 of 16 apps to AA compliance. However, two apps sit at a luminance midpoint where both options produce ratios just below 4.5:1:

| App        | Brand color | Dark fill ratio | White fill ratio |
| ---------- | ----------- | --------------- | ---------------- |
| QuantDrive | `#6366F1`   | 4.23:1          | 4.47:1           |
| QuantAI    | `#8B5CF6`   | 4.46:1          | 4.23:1           |

The code currently uses dark fill for both, yielding 4.23:1 and 4.46:1 respectively. QuantDrive would actually do slightly better with white (4.47:1) but that still falls short. The fundamental issue is that these indigo/purple tones have relative luminance around 0.14–0.18, which produces near-equal contrast against both extremes.

Options: lighten the brand circle fill to `#7C7FFF`/`#A78BFA` (shifts contrast), accept the logotype WCAG exception (SC 1.4.3 exempts logos), or use a different visual treatment for these two apps (outline initial instead of filled text). The v1 review noted this as "decorative brand marks" — if the project commits to that interpretation, a documentation note is sufficient.

## Theme accent and destructive contrast gaps

The test suite validates `foreground`/`background` and (since v2) `primary`/`primaryForeground` for all themes. But the `accent`/`accentForeground` and `destructive`/`destructiveForeground` pairs go untested. Three combinations fail AA:

```
light.accent       #D97706 vs #FFFFFF = 3.19:1
colorblindSafe.accent #EE7733 vs #FFFFFF = 2.87:1
dark.destructive   #EF4444 vs #FFFFFF = 3.76:1
```

The `colorblindSafe` failure is the most concerning — a theme designed for accessibility should not have contrast failures in its own palette. `#EE7733` on white is a warm orange that works well for deuteranopia/protanopia differentiation but fails as a text-on-background pair. If `accentForeground` is meant for text rendered on an accent-colored surface (like a badge or button), this is a functional accessibility bug. If it's only used for large decorative elements, the 3:1 large-text threshold might apply — but the API doesn't distinguish these use cases.

The `dark.destructive` failure is common across design systems (saturated reds on white at small sizes) and could be fixed by switching to `destructiveForeground: '#000000'` for the dark theme, or using a lighter red like `#FF6B6B`.

## Remaining API asymmetry

`generateAppCSS('nonexistent')` returns `/* Unknown app: nonexistent */` and the test explicitly asserts this behavior. `generateThemeCSS('unknown')` and `generateAppIconSet('unknown')` both throw. The fix in this PR addressed the v1 concern by making `generateThemeCSS` throw, but left `generateAppCSS` unchanged because it was pre-existing behavior with an existing test. This is a defensible choice (backward compat), but means consumers still face two error patterns from functions in the same module. A follow-up to align `generateAppCSS` would complete the API consistency.
