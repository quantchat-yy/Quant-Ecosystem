# Unified Data Plane + Legacy API Deletion

Phase 9 removes all nine apps' `api/` folders (~66K LOC of in-memory Map-based service stubs) and replaces them with three shared infrastructure packages (`@quant/data-plane`, `@quant/queue`, `@quant/storage`) plus a `cdc-relay` service that polls the transactional outbox and publishes events to Redpanda/Kafka. The deletion side is clean — the apps already had a `backend/` folder and the `package.json` / `tsconfig` references are updated to point at it. The new packages provide a layered abstraction: BaseRepository handles write-through outbox events, audit logging, field encryption, soft delete, and optimistic locking inside a single Prisma transaction per write.

Watch for:

- **ReplicaRouter.withPrimary is not async-safe** (confirmed) — the `forcePrimary` flag is instance-level mutable state; concurrent async calls will stomp each other's routing decision.
- **Outbox poller sends one Kafka message per event sequentially inside the transaction** (confirmed) — a batch of 100 events produces 100 serial `producer.send()` calls while holding the DB transaction open.
- **Encryption key derived purely from model name** (confirmed) — all records of the same model share the same derived key, making per-user or per-tenant key isolation impossible without reworking `getDerivedKey`.
- **Optimistic lock and the repository's own update double-fetch the record** (confirmed) — when optimistic locking is enabled, `checkAndUpdate` reads + updates, then the repository does its own `findUnique` + `update` on the same row.

---

## High-level view

The data-plane package composes six concerns (read-routing, outbox, audit, encryption, soft-delete, optimistic locking) into a single `DataPlaneRepository<T>` that wraps every write in a Prisma `$transaction`. The design is thin and prescriptive: one transaction per mutation, one outbox row per mutation, one audit row per mutation. This keeps the API surface small but means any caller that needs batching or custom transaction scoping will have to bypass the repository.

The queue package is a typed BullMQ wrapper where Zod validates payloads at enqueue and dequeue time, catching schema drift at job boundaries. The storage package wraps the S3 SDK with a validated config and adds multipart upload via `@aws-sdk/lib-storage`.

The CDC relay is a poll-based outbox consumer. It reads unpublished events, publishes to Kafka one-by-one, then marks the batch published — all inside a single DB transaction. The idempotent producer flag is set, but the per-event serial send pattern means throughput will degrade linearly with batch size and Kafka round-trip latency.

The `EncryptionKey` Prisma model stores per-user encrypted key material but is never referenced by the data-plane package — key management is done entirely via HKDF derivation from a single master key configured at repository construction. This schema/code mismatch signals an incomplete feature.

<summary>Issues (7)</summary>

1. **ReplicaRouter async unsafety** — `withPrimary` mutates instance state across `await` boundaries; use a request-scoped context or `AsyncLocalStorage` instead.
2. **Serial Kafka sends in transaction** — The outbox poller sends one message per event inside `$transaction`; batch them into a single `sendBatch` call before marking published.
3. **Single derived encryption key per model** — `getDerivedKey(modelName)` produces one key for all rows; add a per-record or per-tenant salt to the derivation context.
4. **Double read+update with optimistic locking** — When locking is enabled, the record is fetched and updated in `OptimisticLock.checkAndUpdate`, then fetched again and updated again in the repository's `update` method; merge these into a single path.
5. **Outbox stores encrypted data in payload** — `create` publishes `{ data: encryptedData }` to the outbox, meaning downstream consumers receive ciphertext they cannot decrypt without the model's derived key.
6. **No PrismaClient disconnect in cdc-relay** — `OutboxPoller` creates its own `PrismaClient` but `shutdown()` in `index.ts` only disconnects the Kafka producer; the Prisma connection pool leaks on SIGTERM.
7. **EncryptionKey model unused** — The Prisma schema defines `EncryptionKey` for per-user key storage, but nothing in the data-plane package reads or writes it; the feature is half-wired.

## ReplicaRouter async safety

`ReplicaRouter.withPrimary` sets `this.forcePrimary = true`, runs the callback, then resets it in a `finally` block:

```ts
withPrimary<T>(fn: (client: PrismaClient) => T): T {
  const prev = this.forcePrimary;
  this.forcePrimary = true;
  try {
    return fn(this.primary);   // if fn returns a Promise, finally fires immediately
  } finally {
    this.forcePrimary = prev;
  }
}
```

If two concurrent requests both call `withPrimary(async () => ...)`, the second call's `forcePrimary = true` can be overwritten by the first call's `finally` before the second call's await resolves. The `DataPlaneRepository` itself does not call `withPrimary` (it uses `forWrite()` directly), so this is not currently exploitable through the repository API, but it is a trap for any direct consumer of `ReplicaRouter`.

## Outbox poller throughput and atomicity

The poller's `pollOnce` iterates events one-by-one calling `this.producer.send()` for each:

```ts
for (const event of events) {
  const topic = `outbox.${event.aggregateType}`;
  await this.producer.send(topic, [
    { key: event.aggregateId, value: JSON.stringify(event.payload) },
  ]);
}
```

Each `send` is a full Kafka round-trip. With the default batch size of 100 and typical Kafka latency of 5-10ms, a single poll cycle holds the DB transaction for 500ms-1s. The `KafkaProducerClient` already exposes `sendBatch` which groups multiple topics into a single produce request — the poller should accumulate messages and call `sendBatch` once.

If any single `send` fails mid-loop, the transaction rolls back and no events are marked published — correct for at-least-once delivery — but the poller will re-send all previously successful messages on the next cycle because it has no per-event cursor.

## Field encryption key derivation

All encrypted fields for a given model share the same AES-256-GCM key derived via `HKDF(masterKey, modelName)`. The `EncryptionKey` Prisma model (which stores per-user encrypted keys) is never consulted. If the single master key is compromised, all records across all models are exposed simultaneously. The gap is architectural: the schema models per-user key isolation but the code implements per-model shared keys.

## Double-update in optimistic locking path

When `enableOptimisticLocking` is true and `context.expectedVersion` is provided, the `update` method first calls `optimisticLock.checkAndUpdate(tx, model, id, version, data)` which reads the record, verifies the version, and updates it (incrementing version). Then immediately after, the repository calls `delegate.findUnique({ where: { id } })` and `delegate.update({ where: { id }, data: encryptedData })` — a second read and a second write to the same row. The second update overwrites the version that `checkAndUpdate` just incremented, and it does so without the version guard.

## Outbox payload leaks ciphertext

In `DataPlaneRepository.create`:

```ts
const encryptedData = this.encryptFields(data);
// ...
await this.outbox.publish(tx, ..., `${model}.created`, { data: encryptedData });
```

Downstream consumers (via cdc-relay and Kafka) receive JSON payloads containing AES-256-GCM ciphertext blobs for any encrypted field. These consumers would need access to the same master key and the model name to derive the decryption key — either forcing key distribution to all consumers or making the events useless for consumers that need the plaintext.

## CDC relay Prisma connection leak

`OutboxPoller` instantiates its own `PrismaClient` in the constructor. The `main()` shutdown handler only disconnects the Kafka producer:

```ts
const shutdown = async () => {
  poller.stop();
  await producer.disconnect();
  process.exit(0);
};
```

The Prisma client is never disconnected. In a containerized environment this is largely cosmetic (the process exits), but if the poller is embedded in a longer-lived process or used in tests, the connection pool leaks.

## Test coverage gaps

The test suite (56 tests) mocks all external dependencies. No integration tests exercise the real interaction between the outbox poller and Prisma, or between the queue manager and Redis. The double-update issue with optimistic locking is not caught because the test mocks swallow the second update silently.

Not tested: Prisma migration compatibility of new schema models, Kafka delivery semantics under failure, multipart upload abort/retry, BullMQ retry/backoff configuration.
