# Production crypto and Fastify HTTP layer for Quant auth

This change replaces the toy cryptography in `@quant/auth` (djb2 hashing, `Math.random` tokens, manual base64 JWT encoding) with production-grade libraries: jose for JWT signing/verification, argon2id for password hashing, otplib for TOTP, `crypto.subtle` for PKCE S256, and `node:crypto` for all random values. It also introduces `@quant/server-core`, a Fastify app factory with helmet, CORS, rate limiting, Zod validation, a JWT auth plugin, structured error handling, and health endpoints. QuantMail gains an `api-v2/` directory with Fastify-based identity routes (register, login, refresh, logout, OAuth stubs, OIDC discovery) and a multi-stage Dockerfile.

**Watch for:** Refresh token rotation loses user claims on refresh (confirmed), JWKS key pair regenerates on every process restart with no persistence (confirmed), OAuth `/authorize` endpoint returns authorization codes without any authentication gate (confirmed), in-memory token state makes the service non-scalable and non-restartable (confirmed).

## High-level view

The cryptography swap is thorough. Every `Math.random` call and the hand-rolled djb2 "HMAC" are gone, replaced by `node:crypto` random generators, jose-signed JWTs verified with proper HMAC-SHA256, argon2id with OWASP-tuned parameters, and otplib for TOTP. The change also adds WebAuthn passkey support via `@simplewebauthn/server`, though no routes expose it yet.

The `@quant/server-core` package is a plugin-composition approach: the app factory registers helmet, CORS, rate-limit, cookies, error handling, auth, and health in sequence. Rate limiting falls back to in-memory when Redis is unavailable, which means it operates per-instance in any multi-replica deployment.

The QuantMail api-v2 routes use Zod for request validation, throw errors into Fastify's error handler, and delegate token operations to `TokenService`. The OAuth routes are scaffolding: `/oauth/authorize` issues codes without checking whether the requesting user is authenticated, and the authorization_code grant returns a hardcoded placeholder token.

The Dockerfile runs TypeScript directly in production via `ts-node/esm` rather than compiling to JS, which increases startup time and memory usage. It copies the full `node_modules/` from the builder stage rather than doing a production-only install in the runner.

<summary>Issues (7)</summary>

1. **Refresh rotation drops user claims** — `TokenService.refreshTokens` emits new access tokens with empty email/username and hardcoded `['profile:read']` scopes instead of preserving the original claims. Store original user info and scopes alongside the refresh token family or re-fetch them on refresh.
2. **Ephemeral JWKS key pair** — The RS256 key pair is generated in-memory on first call to `getJWKS()`. Every restart produces a new key, immediately invalidating any federation tokens signed with the previous key. Persist the key pair or rotate with overlapping kid values.
3. **OAuth authorize issues codes without authn** — `/oauth/authorize` returns an authorization code for any request with a valid `client_id` query param. No session or Bearer token check gates the grant. An unauthenticated caller can obtain codes.
4. **In-memory token state is non-durable** — Revocation lists, refresh token families, and active refresh tokens all live in Maps. A process restart clears all revocations and invalidates all outstanding refresh tokens silently.
5. **Readiness probe creates a new Redis connection per request** — `/readyz` instantiates a fresh `ioredis` client, pings, and quits on every call. It would be more robust to share the app-level Redis client and check its status.
6. **Dockerfile runs TypeScript at runtime** — The production CMD uses `node --loader ts-node/esm`, meaning TypeScript is compiled on-the-fly in production. This increases cold-start latency and memory, and ts-node is not designed for production use.
7. **No brute-force / account lockout on login** — The `AuthConfig` defines `maxLoginAttempts` and `lockoutDuration`, but the login route does not track failed attempts or lock accounts. Repeated credential stuffing is unthrottled beyond the global rate limit.

## Token service: refresh rotation drops claims

When a valid refresh token is rotated, the newly minted access token is built with empty strings for email, username, and role, and hardcoded `['profile:read']` scopes:

```typescript
const accessToken = await new jose.SignJWT({
  email: '',
  username: '',
  role: 'user',
  scopes: ['profile:read'] as PermissionScope[],
  app: 'quantmail' as QuantApp,
});
```

Any client that relies on the access token's claims (the auth plugin does, populating `request.auth.email` etc.) will see empty values after their first token refresh. The original claims from `generateTokenPair` are not stored anywhere that `refreshTokens` can retrieve them. Either the refresh token family needs to store the original claim set, or the service needs a user-info lookup path.

## JWKS lifecycle and federation signing

The JWKS key pair is lazily initialized on first call to `getJWKS()` or `signWithPrivateKey()`. Because it lives in a class instance field with no persistence, every process restart (deploy, crash, autoscale) produces a new RS256 key pair with the same `kid: 'quant-primary'`. Any relying party that cached the previous public key will reject tokens signed before the restart. This needs either key persistence (stored encrypted in a database or secrets manager) or key rotation with overlapping validity windows and distinct `kid` values.

## OAuth authorize endpoint: missing authentication gate

The `/oauth/authorize` route validates the query parameters (client_id, redirect_uri, response_type, etc.) but does not check whether the request comes from an authenticated user. In a compliant OAuth 2.0 flow, the authorization endpoint is where the resource owner authenticates and consents. As implemented, any caller can obtain an authorization code by hitting the endpoint with a valid client_id. The code is a `crypto.randomUUID()` that is not stored anywhere, so the token endpoint's `authorization_code` grant returns a hardcoded placeholder token regardless of what code is presented. The scaffolding fails open rather than returning 501 or requiring auth.

## In-memory state: revocation and refresh tokens

`TokenService` tracks revoked tokens, refresh token families, and active refresh tokens in `Map` instances:

- Deploying more than one replica makes revocation ineffective (revoke on instance A, token still valid on instance B).
- A process restart silently un-revokes all tokens and drops all refresh token tracking. Already-issued refresh tokens become invalid because the family data is gone (rejected at the "family not found" check).
- The `cleanup()` method removes revocations older than 24 hours. The current 24h window is safe for 15-minute access tokens but fragile if token lifetimes increase.

The architecture should plan for a shared store (Redis, database) before any multi-instance deployment.

## Readiness probe connection churn

The `/readyz` endpoint creates a throwaway Redis connection on every probe request. This doesn't detect whether the app's _own_ Redis client (used by rate-limit) is healthy, only that a fresh connection can be established. Sharing the app-level Redis instance would provide a more accurate readiness signal and avoid connection churn.

## Dockerfile: runtime TypeScript compilation

The CMD is `node --loader ts-node/esm apps/quantmail/api-v2/app.ts`. The builder stage runs `tsc --noEmit`, confirming the code type-checks, but never emits JS. The runner stage needs ts-node to interpret TypeScript at startup, increasing container cold-start time and bundling the TypeScript compiler into the production image.

The runner stage also copies the full `node_modules/` from the builder rather than running `pnpm install --prod` in the runner. This bundles devDependencies (vitest, pino-pretty, @types/\*) into the production image, inflating its size.

## Backup codes stored unhashed

The `TOTPService.generateBackupCodes()` returns plaintext hex codes. Wherever these are persisted, they should be stored as argon2 hashes (like passwords), since they function as password equivalents. A compromised database backup would expose all backup codes in cleartext.

## WebAuthn challenge storage

The `WebAuthnService` stores challenges in an in-memory `Map<string, string>` keyed by `userId`. Only one challenge per user is stored at a time, so a concurrent registration or authentication attempt from the same user (e.g., two browser tabs) will overwrite the pending challenge, causing the first ceremony to fail silently. The in-memory storage also means challenges don't survive process restarts during the ceremony window.

## Test coverage gaps

Not tested: WebAuthn service (no test file), OAuth token endpoint authorization_code grant, OAuth revoke endpoint, rate-limit behavior, the `cleanup()` method on TokenService, the `needsRehash` upgrade path in a login flow, behavior when Redis is configured and fails.
