# Production crypto and Fastify HTTP layer for Quant auth (v2)

This change replaces toy cryptography in `@quant/auth` with production-grade libraries (jose, argon2id, otplib, crypto.subtle) and introduces `@quant/server-core`, a Fastify app factory with standard middleware. QuantMail gains Fastify-based identity routes and a multi-stage Dockerfile. The v1 review identified 7 issues; 3 have been fixed and 4 were acknowledged as architectural limitations deferred to future work.

**Watch for:** Hardcoded fallback JWT secret in production path (confirmed), duplicate `TokenService` instances across auth and OAuth routes losing shared state (confirmed), Dockerfile compilation fragility with workspace `main` fields pointing to TypeScript source (likely).

## High-level view

The three v1 fixes are correctly implemented. Refresh token rotation now stores user claims alongside the refresh token metadata and re-emits them on rotation, verified by a dedicated test. The readiness probe accepts the app-level Redis client via plugin options and checks its connection status rather than creating throwaway connections. The Dockerfile compiles all packages to JS in the builder stage and runs `node` against the compiled output in a lean production image with non-root user.

The duplicate `TokenService` instantiation pattern in the QuantMail routes is the most impactful remaining concern. Both `routes/auth.ts` and `routes/oauth.ts` instantiate their own `TokenService` via module-level singletons with `getTokenService()`. Because refresh token families and revocation state live in-memory on the instance, a token minted by the auth route's service cannot be refreshed or revoked by the OAuth route's service, and vice versa. This is a correctness bug in the current single-process deployment.

The app configuration uses `process.env['JWT_SECRET'] ?? 'dev-secret-change-in-production'` as a fallback in multiple places. If `JWT_SECRET` is unset in production, the app silently starts with a known secret rather than failing. This is a fail-open pattern for the most critical credential in the system.

1. **Duplicate TokenService instances break token state** — `routes/auth.ts` and `routes/oauth.ts` each create their own `TokenService` singleton. Refresh tokens issued by one cannot be validated or revoked by the other. Share a single instance via Fastify's `decorate` pattern or dependency injection.
2. **Hardcoded JWT secret fallback in production** — Multiple files use `?? 'dev-secret-change-in-production'` for `JWT_SECRET`. If the env var is unset, the app starts with a known-insecure secret. Fail hard (`throw` or refuse to start) when `JWT_SECRET` is missing in production.
3. **Dockerfile tsc depends on workspace package.json `main` fields** — Workspace packages declare `"main": "src/index.ts"`. When the runner copies only `dist/` directories, Node's `require`/`import` resolution won't find the compiled output unless `main` is adjusted or the packages have proper `exports` maps for the compiled layout.
4. **OAuth token endpoint authorization_code grant returns placeholder** — The `authorization_code` path returns `{ access_token: 'placeholder' }`, which is a valid 200 response with a non-functional token. If hit in production, clients receive a token that will fail on every subsequent authenticated request with no indication of why. A 501 status would be safer scaffolding.

## Duplicate TokenService instances across route modules

Both `apps/quantmail/api-v2/routes/auth.ts` and `apps/quantmail/api-v2/routes/oauth.ts` define their own module-level `getTokenService()` factory:

```typescript
// routes/auth.ts
let tokenService: TokenService | null = null;
function getTokenService(): TokenService { ... }

// routes/oauth.ts
let tokenServiceInstance: TokenService | null = null;
function getTokenService(): TokenService { ... }
```

Each produces an independent `TokenService` with its own `activeRefreshTokens`, `refreshTokenFamilies`, and `revokedTokens` maps. A user who registers via `/auth/register` receives a refresh token tracked by the auth route's service instance. If they later use the OAuth `/oauth/token` endpoint with `grant_type=refresh_token`, the OAuth route's instance has no record of that family and will reject the token (or worse, flag reuse detection and revoke it). The same asymmetry applies to `/oauth/revoke` — revoking a token on one instance has no effect on validation in the other.

The fix is straightforward: decorate the Fastify instance with a shared `TokenService` during app creation (in `buildApp()` or via a Fastify plugin), then access it via `request.server.tokenService` in both route modules.

## JWT secret configuration: silent fallback to insecure default

The hardcoded fallback appears in three locations:

- `apps/quantmail/api-v2/app.ts`: `jwtSecret: process.env['JWT_SECRET'] ?? 'dev-secret-change-in-production'`
- `apps/quantmail/api-v2/routes/auth.ts`: same pattern in `getAuthConfig()`
- `apps/quantmail/api-v2/routes/oauth.ts`: same pattern in `getAuthConfig()`

If a deployment forgets to set `JWT_SECRET`, all three resolve to the hardcoded string. The app boots successfully, issues tokens signed with a known secret, and accepts any token signed with that secret. Because the Dockerfile sets `NODE_ENV=production` but doesn't validate required environment variables at startup, this is a realistic deployment failure mode.

The idiomatic fix: validate required config at app startup (in `getConfig()` or `buildApp()`) and throw if critical secrets are missing when `env === 'production'`.

## Dockerfile compilation and module resolution

The Dockerfile builds four packages in sequence:

```dockerfile
RUN pnpm --filter @quant/common exec tsc -b && \
    pnpm --filter @quant/auth exec tsc -b && \
    pnpm --filter @quant/server-core exec tsc -b && \
    pnpm --filter @quant/quantmail exec tsc --outDir dist
```

The runner stage copies `dist/` directories and `package.json` from each package. However, the package.json files declare `"main": "src/index.ts"` (e.g., `@quant/auth` and `@quant/server-core`). When the runner stage resolves `import { TokenService } from '@quant/auth'`, Node follows the `main` field to `src/index.ts`, which doesn't exist in the runner image (only `dist/` was copied). This likely causes a runtime crash on container start.

The fix requires either: (a) overwriting `main`/`types` in each package.json to point to `dist/` during the build stage, (b) copying `src/` alongside `dist/` (defeats the purpose), or (c) using the `exports` field in package.json with conditional exports for built vs source.

## Acknowledged architectural limitations (unchanged from v1)

The following remain as acknowledged future work and are not re-flagged:

- Ephemeral JWKS key pair (needs secrets manager)
- In-memory token state (needs Redis/Prisma backend)
- OAuth authorize without authentication (scaffolding)
- No brute-force lockout (needs persistent state)
