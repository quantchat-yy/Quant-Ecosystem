# Wave 5: Helm chart expansion, feature flags, WebRTC signaling, multi-tenancy and audit

Four new subsystems land in a single wave: Helm chart expansion for all five core apps plus Meilisearch, a feature-flag engine with admin UI, a WebRTC signaling layer extracted from quantmeet, and multi-tenancy (organizations) wired alongside a GDPR-oriented audit/compliance stack. The approach is consistent across all four: each subsystem is a standalone `packages/*` library with in-memory stores, Zod-validated types, Prisma models, server-core plugin glue, and admin UI. The quantmeet WebSocket handler was refactored to delegate to the new `@quant/webrtc` signaling server, replacing inline logic with the shared package.

Watch for: **All in-memory stores silently lose state on restart** (confirmed) — every Wave 5 service (flags, orgs, members, audit) uses in-memory Maps with no persistence path wired, while Prisma models exist but are never referenced by runtime code. **Admin API routes duplicate in-memory stores** (confirmed) — the feature-flags `[id]/route.ts` declares its own `Map` instance, disconnected from the store in `route.ts`, so single-flag GET/PUT/DELETE always return 404. **Org middleware and server-core plugin create independent service instances** (confirmed) — two separate `OrgService`/`MemberService` pairs exist, so data written through one path is invisible to the other. **No auth on admin API routes** (confirmed) — audit, compliance, and feature-flag endpoints accept unauthenticated requests.

## High-level view

The Helm chart additions are structurally sound: proper StatefulSet for Meilisearch with PVC templates, environment-specific overlays, and reasonable resource graduation from staging to production. The chart references helpers that already exist in `_helpers.tpl`, and subchart dependencies for PostgreSQL/Redis are gated behind conditions. The one gap is that the namespace template attempts to create `.Release.Namespace`, which Helm already creates — harmless but redundant.

Feature flags implement a clean domain model (boolean + percentage rollout + rule-based targeting + weighted variants) with deterministic hashing for stable rollout. The `FlagStore` interface is well-abstracted, but nothing bridges it to the Prisma `FeatureFlag` model, meaning the system works only in tests and within a single process lifetime.

WebRTC signaling is well-designed — Zod-validated discriminated union for message types, clean separation between `RoomManager` (room/participant lifecycle) and `SignalingServer` (socket dispatch). The quantmeet integration correctly delegates all protocol handling to the shared package. The client-side `PeerConnectionManager` relies on browser globals (`RTCPeerConnection`, `WebSocket`) without any abstraction boundary, making it untestable outside a browser environment, but that's acceptable for a client module.

The organizations and audit subsystems follow the same in-memory pattern. The audit plugin auto-logs access to sensitive paths, but the path list is hardcoded and the `auth` field access uses unsafe casting. The GDPR compliance admin page is purely cosmetic — it calls mock endpoints that return canned responses rather than connecting to the `DataDeletion`/`DataExporter` classes.

## Disconnected in-memory stores across admin API routes

The feature-flags admin API has a structural wiring bug. `apps/admin/src/app/api/feature-flags/route.ts` declares a module-level `flagsStore` Map with three demo entries. `apps/admin/src/app/api/feature-flags/[id]/route.ts` declares its own _separate_ `flagsStore` Map with zero entries:

```typescript
// [id]/route.ts
const flagsStore: Map<string, Record<string, unknown>> = new Map();
```

Because Next.js route modules are isolated, the `[id]` route's store is always empty. Any `GET /api/feature-flags/flag_demo1` returns 404. The fix is to extract the store into a shared module or, better, wire through the actual `FeatureFlagService`.

## Org middleware vs server-core plugin: two independent instances

The organizations package exports `createOrgContextPlugin` (from `middleware.ts`) and server-core exports `organizationsPlugin` (from `plugins/organizations.ts`). Both instantiate fresh `OrgService()` and `MemberService()` internally:

```
middleware.ts   → new OrgService(), new MemberService()
organizations.ts → new OrgService(), new MemberService()
```

If an app registers the server-core plugin and also registers the middleware plugin separately, or if two different codepaths reference different instances, data written in one is invisible to the other. The server-core plugin decorates `fastify.org.service` and `fastify.org.members`, while the middleware plugin only sets `request.orgContext`. There's no shared instance coordination.

## In-memory stores without persistence

Every Wave 5 package stores state in plain `Map` objects:

- `FeatureFlagService` → `InMemoryFlagStore` (Map)
- `OrgService` → `Map<string, Organization>`
- `MemberService` → `OrgMember[]`
- `AuditLogger` → `AuditEvent[]`

The Prisma schema defines matching models (`FeatureFlag`, `Organization`, `OrganizationMember`, `AuditLog`) with proper indexes, but no code path reads from or writes to them. The `FeatureFlagService.refresh()` method is documented to "re-fetch from database" but is implemented as a no-op. The server-core feature-flags plugin calls `refresh()` on a 30-second interval that does nothing.

This is acceptable for a prototype layer, but the gap between "Prisma models exist" and "runtime uses them" should be explicitly documented as tech debt.

## Admin routes without authentication

All three new admin API routes (`/api/audit`, `/api/compliance`, `/api/feature-flags`) are plain Next.js route handlers with no auth middleware. The compliance route's `POST` handler accepts `delete-data` and `update-retention` actions from any caller — a privilege-escalation vector once these routes connect to real data.

## WebRTC signaling: clean extraction, one lifecycle gap

The extraction from quantmeet to `@quant/webrtc` is well-done. The old inline `handleMessage` switch statement (200+ lines) was replaced with a two-line delegation:

```typescript
signalingServer.handleMessage(participantId, data);
```

The `SignalingServer` validates all messages through a Zod discriminated union before dispatching, which prevents malformed data from reaching room logic.

The one lifecycle concern: `removeConnection` removes the socket from the connections map but does not call `roomManager.leaveRoom()`. If a participant disconnects without sending a `leave` message (browser crash, network drop), they remain as phantom participants in the room indefinitely. Combined with the missing `cleanupStaleRooms` interval, rooms can accumulate ghost participants without bound.

```
disconnect → removeConnection() → socket gone
                                    BUT participant still in room.participants Map
```

## Helm chart: valid structure, Meilisearch security gap

The Meilisearch StatefulSet is correctly templated — PVC templates, health probes, conditional rendering. The production overlay specifies `50Gi` storage and the staging overlay drops to `5Gi` with appropriate resource limits.

The security gap: `MEILI_ENV` is set to `production`, which requires `MEILI_MASTER_KEY` to be set. The values file provides no mechanism to inject this key (no secret reference, no env var from a Secret). Meilisearch will refuse to start in production mode without it.

The namespace template (`templates/namespace.yaml`) creates the release namespace explicitly. This is a known Helm anti-pattern — Helm creates the namespace on install if `--create-namespace` is used, and this template will fail with "already exists" errors in that case. It's harmless if `--create-namespace` is not used, but surprising.

## Audit and retention: well-structured, untested at boundaries

The `RetentionManager.applyRetention` method correctly delegates to `AuditLogger._removeWhere`, and the tests verify the date-boundary logic by manually backdating timestamps. The `DataDeletion` class logs its own deletion event after removing user data, creating an audit trail for GDPR compliance.

The `_removeWhere` and `_getEvents` methods are prefixed with underscores to signal internal use, but they're public and exported. The retention tests directly mutate event timestamps via casting (`(event as { timestamp: Date }).timestamp = oldDate`), which works but wouldn't survive a persistence layer where timestamps are server-generated.

Not tested: `DataDeletion.scheduleDataDeletion` execution (only scheduling is tested, not the execution path), `DataExporter.generateExportManifest` size estimation, and the audit plugin's path-matching logic.

## Test quality assessment

The test suites cover the core happy paths and important edge cases (room-full, duplicate-join, flag percentage determinism, retention date boundaries). The signaling server tests verify all message types and error paths. The WebRTC test mock (`createMockSocket`) is minimal but sufficient for verifying dispatch behavior.

Not tested: the `PeerConnectionManager` client (relies on browser globals, no test infrastructure exists for it), the admin API routes (no integration tests), the organizations middleware plugin (the test only covers `OrgService` directly, not the Fastify plugin registration), and concurrent access patterns across any in-memory store.
