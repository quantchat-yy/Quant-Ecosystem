# Universal Notification Center and Yjs Collaboration Server

Phase 27 adds a full delivery pipeline to the existing notification center: DND enforcement, 5-minute batch windows, cross-device dedup, snooze/remind-me, VAPID Web Push, per-thread mute, preview privacy, inline reply routing, and important-only mode. Phase 28 extends the Yjs collaboration server with Postgres persistence, S3 snapshots, awareness protocol (cursors/selections), per-paragraph permissions, version history with named checkpoints, experimental doc branching (fork + merge via CRDT state vectors), a Yjs-backed infinite-canvas whiteboard, Y.Text code editing, and mention-triggered notifications. All state lives in-memory Maps, with persistence and storage behind injectable adapter interfaces.

**Watch for:** DND schedule matching ignores the configured timezone (confirmed). The `sendWithPipeline` dedup check uses the freshly-generated `id` — which can never be a duplicate — rendering cross-device dedup inert in that path (confirmed). The `diffCheckpoints` function only computes net character-count delta, not a meaningful diff (confirmed). Batch windows inside `UniversalNotificationCenter.addToBatch` are never flushed by a timer — they only flush when the _next_ notification arrives past the window boundary (confirmed).

## High-level view

The notification delivery pipeline is split into two layers: a set of standalone services (`DndService`, `BatchService`, `DedupService`, `SnoozeService`, `WebPushService`) and a `sendWithPipeline` method on `UniversalNotificationCenter` that reimplements DND, dedup, thread mute, and important-only inline rather than composing the standalone services. The standalone services are well-factored, but their relationship to the center's pipeline is unclear — callers must decide which to use.

The Yjs server integration takes an options-based composition approach: `YjsServerOptions` injects persistence, storage, awareness, permissions, version history, and branching as optional adapters. This is clean for testing and keeps the server backward-compatible (no arguments = in-memory only). Periodic flush uses `setInterval` with `.unref()`, and the server properly shuts down timers on disconnect and via `shutdown()`.

Permission enforcement operates at the `applyUpdateWithPermissions` call site, requiring callers to pass both a `paragraphId` and a `userId`. Since a single Yjs update can span multiple paragraphs, this per-call enforcement may be trivially bypassed if a client packs edits to multiple regions into one update.

Doc branching uses Yjs state-vector diffing for merge, which is correct for non-conflicting concurrent edits but offers no conflict reporting or merge-conflict UI surface. The whiteboard service manages its own `Y.Doc` instances separate from the main `YjsServer`, meaning persistence and eviction are not applied to whiteboard documents.

<summary>Issues (7)</summary>

1. **DND ignores configured timezone** — `DndService.isInDndWindow` and `UniversalNotificationCenter.isDndActive` use `new Date()` local time without converting to the user's `config.timezone`. Users in different timezones will have DND apply at the wrong hours. Apply timezone offset before comparing.
2. **Pipeline dedup is no-op** — `sendWithPipeline` generates a fresh `id`, then checks `deliveredIds` for that id. It will never find it. Either dedup by a content-derived key (type+sender+thread+recipient) or accept the notification id from the caller.
3. **Batch windows lack timer-based flush** — `addToBatch` only flushes when a new notification arrives after the window expires. If the last notification in a batch arrives and no subsequent one comes, the batch is never delivered until `flushBatches` is called externally. Document or implement a periodic flush.
4. **Per-paragraph permission granularity mismatch** — A single Yjs update can span multiple paragraphs. `applyUpdateWithPermissions` takes one `paragraphId`, so multi-paragraph updates bypass permission checks for non-specified paragraphs. Consider parsing the update to extract affected ranges, or document this limitation.
5. **Whiteboard docs bypass persistence/eviction** — `WhiteboardService` creates its own `Y.Doc` instances that are invisible to `YjsServer`. They are never flushed to Postgres or S3, and never evicted. Route whiteboard docs through `YjsServer` or give `WhiteboardService` its own persistence path.
6. **diffCheckpoints only counts net character delta** — `VersionHistoryService.diffCheckpoints` computes `max(0, lenB - lenA)` for added/removed. A change that deletes 10 chars and adds 10 different chars reports `{addedChars: 0, removedChars: 0}`. Use a real diff algorithm or rename to clarify it's a size delta.
7. **Standalone services vs. center pipeline duplication** — `DndService`, `DedupService`, and batch logic exist as standalone classes but are reimplemented differently inside `UniversalNotificationCenter`. Callers need guidance on which to use. Consider having the center compose the services internally.

## DND timezone gap

Both `DndService` and `UniversalNotificationCenter.isDndActive` store a `timezone` field on the config but never use it. The time comparison runs against `new Date()` which uses server-local time:

```typescript
// DndService.isInDndWindow
private isInDndWindow(config: DndConfig, now: Date): boolean {
  const currentDay = now.getDay();           // server-local
  const currentMinutes = now.getHours() * 60 + now.getMinutes(); // server-local
  ...
}
```

A user configured for `America/Los_Angeles` whose server runs in UTC will have DND trigger 7-8 hours too early. The fix is to convert `now` to the user's timezone before extracting day/hours/minutes. This affects both the standalone `DndService` and the duplicated logic in `UniversalNotificationCenter`.

## sendWithPipeline dedup is structurally inert

The dedup step in `sendWithPipeline` checks whether `full.id` has been seen before — but `full.id` is generated two lines above with a unique counter:

```typescript
const id = `notif_${Date.now()}_${++this.counter}`;
const full = { ...notification, id, timestamp: Date.now(), read: false };
...
const userDelivered = this.deliveredIds.get(userId) ?? new Set();
if (userDelivered.has(full.id)) { return null; } // always false
```

The standalone `DedupService` correctly operates on an externally-supplied `notificationId`, so the dedup concept works when using that service directly. The center's pipeline implementation just doesn't achieve the stated goal.

## Batch window delivery gap

`addToBatch` accumulates notifications and only flushes when a _new_ notification arrives after the window boundary. If a user receives 3 notifications in a batch window and then silence, those 3 notifications sit in the map indefinitely until `flushBatches()` is called. Neither a timer nor a scheduler drives that call. The standalone `BatchService` has `flushExpired()` with the same semantics — it must be polled externally. The test suites verify the happy path (next-notification-triggers-flush) but don't exercise the stale-batch scenario.

## Per-paragraph permission enforcement surface

`applyUpdateWithPermissions(docId, update, userId, paragraphId)` checks a single paragraph's permission before applying the raw Yjs binary update. A Yjs update is opaque at this level — it can contain changes to any number of shared types. If a client sends one update that touches paragraphs A and B, calling `applyUpdateWithPermissions` with `paragraphId = A` will bypass the check for B. This is a fundamental tension with CRDT binary updates; real enforcement would need to operate at the Y.Doc observation layer (e.g., `doc.on('update')` with origin-based authorization), not as a gate before `Y.applyUpdate`.

## Whiteboard isolation from server lifecycle

`WhiteboardService` maintains its own `docs: Map<string, Y.Doc>` independent of `YjsServer.docs`. Consequences:

- Whiteboard docs are never flushed to Postgres or snapshotted to S3.
- Whiteboard docs are never evicted — they accumulate in memory indefinitely.
- Awareness and version history don't apply to whiteboard docs (separate state maps).

Similarly, `CodeCollabService` manages its own `Y.Doc` instances. If these are intended as standalone in-memory services for now, the limitation should be documented. If they should participate in persistence, they need to route through `YjsServer` or accept the same adapter options.

## Version history diff fidelity

`VersionHistoryService.diffCheckpoints` reconstructs two docs, extracts their `'content'` Y.Text as strings, and computes:

```typescript
const addedChars = Math.max(0, textB.length - textA.length);
const removedChars = Math.max(0, textA.length - textB.length);
```

This is a length delta, not a diff. Replacing "hello" with "world" (same length) produces `{added: 0, removed: 0}`. The method name and return type (`CheckpointDiff`) imply more than it delivers. Either implement a character-level diff (e.g., Myers) or rename to `sizeComparison` to set correct expectations.

## Test coverage gaps

Tests cover the happy paths thoroughly (241 total tests passing). Notable gaps:

- No test exercises `DndService` with timezone offset (because the feature doesn't work).
- No test exercises `sendWithPipeline` attempting real cross-device dedup (because it can't trigger).
- No test verifies that `WhiteboardService` or `CodeCollabService` docs integrate with persistence.
- No test for `addToBatch` with a stale window that never receives a subsequent notification.
- The `applyUpdateWithPermissions` test in paragraph-permissions.test.ts doesn't verify multi-paragraph updates.
