# Unified identity, wallet, and permission model across 13 apps (v2)

Phase 24 introduces a platform-wide ABAC+RBAC permission engine, extends auth with cross-app SSO and federated "Sign in with Quant" for third-party developers, adds a unified wallet with India-market payment gateways (Razorpay, UPI), and layers on a single Quant Pro subscription with Apple/Google IAP validation. All seven issues raised in v1 have been addressed with structural fixes. The remaining concerns are second-order: test coverage gaps around the newly-added validation paths, and a residual ABAC bypass via empty-context objects.

**Watch for:** ABAC conditions pass when context is `{}` because individual attribute-undefined checks return true, allowing callers to bypass conditions by providing an empty context object instead of omitting it entirely (confirmed). The SDK's clientSecret validation is only enforced for pre-registered clients, meaning unregistered clients still exchange tokens without secret checks (confirmed). The `PurgeableService` coordination mechanism has zero test coverage (confirmed).

**Verdict**: APPROVED

## High-level view

The permission engine now correctly denies access when policies have conditions and `context` is entirely omitted (undefined). However, `evaluateCondition` returns `true` when a specific attribute is absent from the provided context object, which means `can(subject, action, resource, {})` still bypasses all ABAC conditions. This is a deliberate open-world assumption documented in the test ("passes when context attribute is not provided"), but it creates a semantic gap: the engine protects against forgotten context parameters but not against empty context objects.

The SDK's clientSecret fix uses a conditional path: validation only fires when the client is pre-registered via `sdk.registerClient()`. For unregistered clients (i.e., the happy path in all existing tests), any secret value is accepted. No test exercises the rejection path for wrong secrets on registered clients, so the validation code has zero coverage.

Account purge now iterates over registered `PurgeableService` instances, but no integration test actually registers a purgeable service and verifies that `purgeUserData` is called. The existing deletion-gate test manually calls `sessionService.revokeAllSessions()` outside of `purgeAccount()`, which doesn't validate the coordination contract.

1. **ABAC empty-context bypass** — `can(subject, action, resource, {})` passes all conditions because `evaluateCondition` returns `true` when the attribute isn't in the context object. A caller who passes an empty context instead of omitting it entirely bypasses all ABAC gates. 2. **SDK secret validation only covers registered clients** — `handleCallback` and `refreshToken` skip secret validation entirely when `registeredClients.get(clientId)` returns undefined. No test exercises the secret-rejection path. Add a test that registers a client, then attempts token exchange with a wrong secret.
3. **PurgeableService coordination untested** — `AccountLifecycleService.purgeAccount` iterates `purgeableServices` but no test verifies this. The deletion-gate test manually orchestrates cleanup outside of `purgeAccount`. Add a test that registers a mock purgeable service and asserts `purgeUserData` is called.

## ABAC empty-context object still passes conditions

The v1 fix correctly handles the case where `context` is `undefined`:

```typescript
if (policiesWithConditions.length > 0) {
  if (!context) {
    return false; // Fixed in v2: denies when context omitted entirely
  }
  // ...evaluates conditions
}
```

But `evaluateCondition` has a separate permissive path:

```typescript
private evaluateCondition(condition: ABACCondition, context: ABACContext): boolean {
  const contextValue = context[condition.attribute];
  if (contextValue === undefined) {
    // If context doesn't provide the attribute, condition cannot be evaluated - pass
    return true;
  }
  // ...
}
```

Calling `engine.can(subject, 'write', 'wallet', {})` passes the `if (!context)` check (context is truthy), then every condition evaluates to `true` because no attributes are present. The test suite explicitly validates this behavior (`'passes when context attribute is not provided (permissive)'`), so it's a deliberate choice. The risk is that it's easy for a caller to construct a minimal context object (`{}`) and unintentionally bypass all restrictions. A safer default would be to deny when a required attribute is missing from context, or at minimum log it. (confirmed)

## SDK secret validation conditional on pre-registration

```typescript
const registeredClient = this.registeredClients.get(clientId);
if (registeredClient) {
  if (registeredClient.clientSecret !== clientSecret) {
    return null;
  }
}
```

The entire test suite never calls `sdk.registerClient(...)` before exercising `handleCallback`. Every test passes arbitrary strings as `clientSecret` and succeeds because the lookup returns undefined. The secret-validation code path is never exercised by any test, leaving it possible for a regression to go undetected. (confirmed)

## SSO scope handling remains permissive-then-pass-through

`validateCrossAppToken` checks that at least one token scope matches the target app's allowed list, then returns the full payload including all original scopes. Downstream code receives a token validated for the target app but carrying scopes like `agent:execute` that the target app doesn't allow. The middleware doesn't downscope the payload. This was not in the v1 fix list and remains a design observation rather than a blocking issue. (likely)
