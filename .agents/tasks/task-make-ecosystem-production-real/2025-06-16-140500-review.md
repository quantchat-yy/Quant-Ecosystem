# Production wiring across agentic, storage, federation, ML, and build layers

The ecosystem moves from simulated stubs to real backends across five dimensions: agent tools now call live HTTP services, file storage delegates to S3, the Matrix bridge connects via matrix-bot-sdk, ML classifiers delegate to Qdrant/ONNX/AI-SDK, and the SFU service routes through LiveKit. Every integration follows the same dual-mode pattern: detect config from environment variables, use the real backend when configured, fall back to the existing in-memory implementation otherwise. Build configs are corrected for four apps, and graceful shutdown is added to two services.

Watch for: fire-and-forget async in the SFU service loses errors and creates race conditions where callers read stale state (confirmed); the upload route does not sanitize the filename extension before constructing the S3 key, allowing arbitrary extensions to reach the object store (confirmed); the Planner's LLM integration parses JSON from an LLM without structural validation, and a malformed response will surface a raw parse error to callers (confirmed).

## High-level view

The agentic layer gains an HTTP client and rewires all five agent classes to POST/GET against their respective app backends, with graceful fallback on failure. The agents pass user-supplied parameters (potentially LLM-generated) to backend APIs without any client-side schema validation.

The QuantDrive storage service replaces the placeholder file upload with real multipart handling via `@fastify/multipart` and delegates to `@quant/storage` for S3. The local-filesystem fallback uses `path.basename` for key-to-filename resolution (safe against traversal), but the upload path trusts the client-supplied extension in the S3 key.

The federation layer upgrades MatrixBridgeBot to use matrix-bot-sdk with dynamic import and auto-join mixin, and introduces a `RoomMappingStore` abstraction for pluggable persistence. The bot has no reconnection logic for dropped connections — `isConnected()` returns stale true after a disconnect.

The ML pipeline introduces backend interfaces (VectorStoreBackend, SpamModelBackend, SentimentBackend, NERBackend) on every classifier, keeping the in-memory implementations as fallbacks. All backend errors are silently swallowed, meaning misconfigured providers degrade to fallback mode with no observability signal.

The SFU service's LiveKit integration uses fire-and-forget promises for room creation and token generation that swallow errors and leave internal state inconsistent on failure.

The Vite override is bumped to `>=6.4.3` (CVE-2025-31486 fix) and form-data to `>=4.0.6` (CRLF injection fix), but both use open-ended ranges that could pull future majors.

1. **SFU fire-and-forget race** — `createTransport` and `connectTransport` kick off promises for room creation and token generation without awaiting. Callers can read `livekitToken` or `livekitRooms` state before the promise resolves, and the `.catch(() => {})` handler silently drops errors that should be propagated or retried.
2. **Upload filename extension passed to S3 key unsanitized** — `path.extname(filename)` on a user-controlled filename means the stored S3 key inherits whatever extension the client sends (e.g., `.html`, `.exe`). If signed URLs are later served without Content-Disposition, the browser may interpret them as executable content.
3. **Planner JSON parse without validation** — `JSON.parse(response.content)` on LLM output is unguarded. A malformed response throws a `SyntaxError` that propagates to the caller. There is no zod/schema validation on the parsed shape, so a structurally incorrect response silently produces invalid plans.
4. **Matrix bridge lacks reconnection** — If the Matrix homeserver drops the connection after `start()`, no reconnection logic re-establishes it. `isConnected()` stays true (since `started` is never reset on disconnect), so messages are lost.
5. **SFU room creation error leaves room in limbo** — The catch handler on `createRoom` calls `this.livekitRooms.delete(roomId)`, but a second `createTransport` for the same room may arrive between the initial set('pending') and the delete, causing both to attempt room creation.
6. **Memory store embedding failures silently ignored** — `store()` catches and swallows embedding generation errors. If the embedding provider is misconfigured, all stored memories silently lack embeddings and vector search returns nothing, with no indication to the caller.
7. **Vite override uses open range** — The `form-data` override (`>=4.0.6`) and `vite` override (`>=6.4.3`) use open-ended ranges that could pull in future major versions with breaking changes.

## SFU LiveKit integration: fire-and-forget concurrency

The SFU service's `createTransport` and `connectTransport` methods kick off async operations (room creation, token generation) without exposing the resulting promises or awaiting them:

```typescript
// In createTransport:
this.livekitRooms.set(roomId, 'pending');
this.livekit.createRoom(roomId)
  .then((room) => { this.livekitRooms.set(roomId, room.sid); })
  .catch(() => { this.livekitRooms.delete(roomId); });
```

This creates a window where `livekitRooms.get(roomId)` returns `'pending'` and subsequent callers assume the room exists. If creation fails, the room is deleted, but between the `.set('pending')` and the `.delete()`, another `createTransport` call won't attempt to re-create the room (the `!this.livekitRooms.has(roomId)` guard already passes).

The token generation in `connectTransport` has the same pattern: `transport.livekitToken` is undefined until the promise resolves, and any caller of `getParticipantToken` immediately after `connectTransport` gets `undefined` with no indication that a token is in-flight.

The `.catch(() => {})` handlers throughout mean operational failures (auth misconfiguration, network partition to LiveKit) produce no observable error — the system silently degrades without any signal to observability.

## Upload route: filename extension reaches S3 key

In `storage.service.ts`, the S3 key is constructed as:

```typescript
const ext = path.extname(filename); // user-controlled
const key = `uploads/${fileId}${ext}`;
```

The `fileId` is a UUID (safe), but `ext` is whatever the client sent in the multipart filename header. This means an attacker can store objects with extensions like `.html` or `.svg`. If the application later generates signed S3 URLs for direct browser access without forcing `Content-Disposition: attachment`, a stored XSS vector exists. The upload route also does not restrict content types — any MIME can be uploaded up to the 100 MB limit.

## Planner LLM JSON parse lacks schema validation

The planner sends a structured prompt to the LLM and parses the response with bare `JSON.parse`:

```typescript
const parsed = JSON.parse(response.content);
const steps: PlanStep[] = (parsed.steps || []).map((s: any, i: number) => ({ ... }));
```

If the LLM returns markdown-wrapped JSON (common), or a response with extra text outside the JSON, `JSON.parse` throws. The outer try/catch in `llmPlan` catches this and falls through to keyword planning, but the error is swallowed — no warning is logged, and the system silently degrades from LLM planning to keyword heuristics without observability. Additionally, there is no validation that `parsed.steps` contains objects with the expected shape; if any field is wrong type, the resulting `PlanStep[]` silently carries garbage.

## Matrix bridge connection lifecycle

The bridge bot connects once in `start()` and sets `this.started = true`. The matrix-bot-sdk's `MatrixClient` emits lifecycle events (sync failure, connection drop), but the bridge does not subscribe to any of them. If the homeserver becomes unreachable after initial connection:

- `isConnected()` still returns `true`
- `sendToMatrix()` will throw, caught by the `.catch()` in `onQuantMessage`
- No reconnection is attempted
- Messages are dropped until the process restarts

The `stop()` method exists for graceful shutdown but is never called from within the class on error.

## Agent HTTP tools: no input validation on forwarded parameters

All five agent classes forward parameters directly to their backend HTTP endpoints:

```typescript
execute: async (params: any) => {
  const response = await this.httpClient.post('/api/emails', {
    to: params.to,
    subject: params.subject,
    body: params.body,
    attachments: params.attachments || [],
  });
```

The `params` object comes from tool invocation by the planner (which may be LLM-driven). There is no validation of parameter shapes or types before they reach the HTTP layer. The lack of client-side validation means the system relies entirely on each backend's individual input validation — and if the planner hallucinates a malformed tool call, the error path is exercised on every request.
