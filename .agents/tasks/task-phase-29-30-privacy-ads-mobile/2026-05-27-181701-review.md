# Privacy-first ad targeting, creator economy, and Capacitor mobile shell (v2)

This is a re-review after fixes were applied to the six critical issues from v1. The change adds an on-device ad ranker with privacy enforcement, creator monetization services (pay-per-view, storefronts, compute credits, tax documents), a Capacitor mobile shell with 10 native plugins, and integration wiring into the quantads backend. All six production-readiness issues from v1 have been addressed: consent persistence now has a `ConsentStore` interface, revenue uses integer-cent arithmetic, interest updates are category-scoped, inbound request validation is active, brand safety uses word-boundary regex, and the offline sync executor is injectable.

**Watch for:** The ads-api service still instantiates `PrivacyEnforcerService` without wiring it into any request pipeline — it validates startup availability only (confirmed). The `BehavioralOptInService` sync/async duality (`getConsent` vs `getConsentAsync`) silently degrades async stores to `false` without warning (confirmed). OAuth remains mock-only with no path to real token exchange (confirmed, acknowledged as intentional for this phase).

## High-level view

The privacy-ads package now enforces privacy bidirectionally: inbound requests are validated for tracking cookies before candidates are served, and outbound responses are audited against the tracking-domain blocklist. The `updateLocalInterests` method correctly scopes weight adjustments to matched categories via a `contextCategories` field on feedback, with a fallback to all-categories when the field is absent (backward compat).

Revenue arithmetic across `PayPerViewService` and `StorefrontService` accumulates in integer cents via a parallel `revenueCents` Map and converts to dollars only on read.

The `BehavioralOptInService` accepts an optional `ConsentStore` interface, defaulting to `InMemoryConsentStore`. The interface supports both sync and async backends, though the primary `getConsent()` path silently returns `false` for async stores — a subtle hazard for production integrators who inject a database-backed store without noticing they need `getConsentAsync`.

The offline sync's `replayOnReconnect` now delegates to a configurable `MutationExecutor` callback injected via constructor. The default remains `async () => true` (safe for tests), but the injection point allows production code to wire real network calls.

The ads-api standalone service (`services/ads-api/src/main.ts`) still only proves that `PrivacyEnforcerService` can be instantiated. It logs "Privacy-first ad enforcement: ACTIVE" without mounting any routes or intercepting requests — potentially misleading to operators.

## ConsentStore async/sync API hazard

The `ConsentStore` interface declares methods that return `boolean | undefined | Promise<boolean | undefined>`:

```typescript
export interface ConsentStore {
  get(userId: string): boolean | undefined | Promise<boolean | undefined>;
  set(userId: string, consented: boolean): void | Promise<void>;
}
```

The synchronous `getConsent()` method checks `if (value instanceof Promise) { return false; }`. An async store injected by a production integrator silently fails closed (returns `false`, denying behavioral targeting) without any log or metric. While fail-closed is the safe default from a privacy perspective, it's operationally invisible. The `getConsentAsync` method exists but nothing in the codebase guides integrators toward it — no deprecation notice on the sync method, no runtime warning when a Promise is detected.

## ads-api misleading startup log

```typescript
const enforcer = new PrivacyEnforcerService();
void enforcer.getStrictCSPDirectives();
// ...
console.log('Privacy-first ad enforcement: ACTIVE');
```

The log message implies runtime enforcement is running. An operator seeing this in production logs would reasonably assume inbound ad requests are being validated. The actual enforcement lives exclusively in `apps/quantads/backend/routes/privacy-ads.ts`. Either the log should say "Privacy-ads package validated (enforcement via quantads routes)" or the service should mount its own enforcement middleware.
