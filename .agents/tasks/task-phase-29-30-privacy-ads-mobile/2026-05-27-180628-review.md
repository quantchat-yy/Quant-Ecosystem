# Privacy-first ad targeting, creator economy extensions, and Capacitor mobile shell

This change adds four capabilities across three new/extended packages: an on-device ad ranker with contextual-first targeting and hard privacy enforcement (`packages/privacy-ads/`), creator monetization services including pay-per-view, storefronts, compute credits, and tax documents (`packages/payments/`), a Capacitor mega-shell mobile app wrapping 10 native plugins with OAuth, deep linking, offline sync, crash reporting, and performance budgets (`apps/quant-mobile/`), plus integration wiring the privacy-ads package into the quantads backend and ads-api service. The privacy model defaults to contextual-only targeting with behavioral as opt-in, and the PrivacyEnforcerService validates outgoing responses against known tracking patterns.

**Watch for:** Consent state lives only in-memory with no persistence layer, meaning a service restart resets all user opt-in decisions (confirmed). The OAuth implementation returns mock tokens without actually performing the PKCE exchange against a real authorization server (confirmed). The privacy enforcer's pattern-matching approach will miss novel trackers not on its hardcoded list (likely). Floating-point arithmetic in revenue calculations produces precision drift for aggregated values (confirmed, noted in FEAT-002 findings).

## High-level view

The privacy-ads package establishes a clear separation: the server sends 50 candidate ads with no user-identifying data, the client ranks locally using an on-device interest model, and only aggregate feedback (clicked/dismissed) flows back. The enforcement layer validates outgoing payloads against known tracking domains and cookie patterns, with strict CSP directives blocking frames, mixed content, and external connections. The design relies entirely on a static blocklist that requires ongoing maintenance to remain effective.

Creator economy extensions follow the existing payments service pattern: in-memory Maps as data stores, Zod schemas for input validation, and an 85/15 revenue split for paywalls and storefronts. The E2E test exercises the full earn-to-tax-document lifecycle. Revenue accumulation uses floating-point addition which drifts at scale; the test works around it with `toBeCloseTo` rather than fixing the arithmetic.

The mobile mega-shell is a TypeScript-first Capacitor project that defines plugin wrappers, auth flows, and offline sync integration. All native platform interactions are stubbed with mock implementations. The offline sync delegates to the existing `@quant/sync-engine` OfflineOperationQueue but its replay callback is a no-op that marks all operations as successful without executing them.

The integration layer wires privacy-ads into the quantads Fastify backend with three routes and validates ad _responses_ before serving, but never validates inbound _requests_ for tracking cookies. The ads-api service instantiates a PrivacyEnforcerService at startup but does nothing with it beyond calling `getStrictCSPDirectives()` to avoid an unused-variable error.

## On-device ranker and interest model corruption

The `OnDeviceRankerService.updateLocalInterests()` method applies feedback uniformly across all interests in the model. When a user clicks ad_1 (matched via "tech" category), the weight for every interest (tech, sports, food, etc.) increases by 0.1. This makes the interest model converge toward a flat distribution over time rather than reflecting actual user preferences. The correct behavior would be to identify which interest categories matched the clicked ad and update only those weights.

```typescript
// Current: updates ALL interests on any feedback
const updatedInterests = model.interests.map((interest) => {
  if (feedback.action === 'clicked') {
    return { ...interest, weight: interest.weight + 0.1, lastSeen: feedback.timestamp };
  }
  // ...
});
```

The test suite verifies that weights increase/decrease but doesn't test that only the relevant category is affected.

## Privacy enforcement gaps in the serving layer

The privacy-ads route in `apps/quantads/backend/routes/privacy-ads.ts` accepts requests without checking inbound headers for tracking cookies. The `PrivacyEnforcerService.validateRequest()` exists for exactly this purpose but is never called in the request lifecycle.

The `PrivacyAdServingService` does call `this.privacyEnforcer.auditAdResponse(candidates)` before returning candidates, which prevents tracking URLs from appearing in responses. But the asymmetry means inbound tracking signals are silently accepted, potentially enabling correlation by a network observer who can see both the cookie-bearing request and the candidate response.

## BehavioralOptInService has no persistence contract

The consent store is a plain `Map<string, boolean>()` with no interface for injecting a backing store. In production:

- Process restart resets all consent to `false` (the default)
- Multiple service replicas cannot share consent state
- No audit trail of when consent was granted or revoked

The default (false) is the safe fallback, so losing consent isn't a data leak, but it is a UX regression: users who opted in get silently downgraded. The service needs a `ConsentStore` interface parameter.

## Revenue arithmetic without integer precision

`PayPerViewService` and `StorefrontService` both accumulate revenue with `paywall.revenue += creatorRevenue`. While individual calculations use `Math.round(... * 100) / 100`, repeated additions on IEEE 754 floats accumulate errors. The FEAT-002 findings acknowledge this (`507.45000000000005`). For a payments system, store amounts in integer cents and divide only for display.

## OAuth: PKCE generation correct but unintegrated

`signInWithApple()` and `signInWithGoogle()` call `this.generatePKCE()` but discard the result, returning mock tokens instead. The PKCE value is never sent to an authorization endpoint or stored for the token exchange step. The class's public API makes it appear complete, which could mislead integrators into believing auth is functional.

## Brand safety keyword matching produces false positives

The `BrandSafetyService` uses `string.includes()` for keyword detection. The "kill" keyword in the violence category matches "skill", "killjoy", etc. "hate" in hate_speech matches "whatever". For production this needs word-boundary matching (`\bkill\b`) or an NLP classifier.

## Mobile offline sync: replay executor is a no-op

`MobileOfflineSync.replayOnReconnect()` passes `async () => true` to `this.queue.replayAll()`, meaning every queued operation is marked as successfully replayed regardless of whether the actual mutation was applied server-side.

```typescript
const result = await this.queue.replayAll(async () => {
  return true; // always "succeeds" without doing anything
});
```

The service needs a callback parameter or injected executor that performs the actual network request for each queued operation.

## Test coverage gaps

Not tested: deep-link handler URL parsing edge cases (double slashes, query params, fragments), brand safety false-positive scenarios, concurrent access on shared Maps, the actual Fastify route layer (tests hit services directly), and the crash reporter under high-frequency breadcrumb ingestion.
