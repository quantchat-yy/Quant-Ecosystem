# Universal search orchestrator with RAG synthesis and cross-collection similarity

Phase 25 adds the user-facing universal search pipeline: a main orchestrator that wires NL query enhancement, hybrid BM25+vector search, post-query permission filtering, snippet highlighting, and optional RAG-based AI answer generation into a single `search()` call. Supporting services include per-user search history with incognito, a content extractor for PDFs/images/video transcripts, cross-collection "find similar" via vector nearest-neighbor, and a typeahead service aggregating recent/popular/autocomplete suggestions. An HTTP router in the search-indexer service exposes five endpoints backed by these services.

**Watch for:** The router hardcodes `isAdmin: false` on every request, meaning admin users cannot exercise elevated permissions through this API (confirmed). The universal search orchestrator runs scopes sequentially rather than concurrently, which will serialize latency across collections (confirmed). The RAG synthesizer has no error handling around the AI engine call — a provider timeout or rejection will propagate as an unhandled exception and fail the entire search request even for non-AI results (confirmed).

## High-level view

The orchestrator accepts seven injected services and runs an eight-step linear pipeline. Permission filtering happens post-query on the merged result set — the correct defense-in-depth position. The permission model in the router is incomplete: every request is stamped `isAdmin: false` regardless of the caller's actual role, so the entire admin-access path tested at the service layer is unreachable from the API.

The RAG synthesizer rejects hallucinated citations (IDs must exist in the provided context). The gap is operational — there's no timeout, no fallback, and no circuit-breaker around the AI call, so a slow or failing LLM provider degrades the entire search experience rather than just the AI answer.

The find-similar service correctly searches collections concurrently and swallows per-collection errors. The orchestrator's scope search does not — it awaits each scope sequentially, serializing what should be parallel work.

The content extractor is a v1 stub: PDF extraction uses heuristic BT/ET marker parsing, image OCR returns empty. The PDF fallback path (4+ printable character runs) will index PDF structure keywords and font names as document text. (likely)

The `page` parameter is accepted in the schema but never applied — all callers receive page 1 regardless of what they request.

<summary>Issues (5)</summary>

1. **Router hardcodes isAdmin: false** — `UniversalSearchRouter.search()` always passes `permissions: { userId, isAdmin: false }`. Admin callers cannot see private documents through this API. Either accept the role from the authenticated session or add an `isAdmin` field to the request schema with proper auth gate.
2. **RAG synthesis has no error boundary** — If `aiEngine.infer()` rejects or times out, the entire `UniversalSearchService.search()` call fails. The non-AI search results are already computed but never returned. Wrap the RAG call in a try/catch and return results without `ragAnswer` on failure.
3. **Sequential scope search serializes latency** — `UniversalSearchService.search()` iterates scopes with a `for...of` loop, awaiting each one. With 5+ scopes this multiplies wall-clock time. Use `Promise.all` (same pattern as `FindSimilarService`).
4. **No pagination offset** — The `page` option is accepted but never used in the orchestrator. Results are sliced `0..limit` regardless of page number. Either implement `offset = (page - 1) * limit` or remove the field from the schema to avoid misleading callers.
5. **Snippet XSS if rendered in HTML** — Highlighted snippets inject `<mark>` tags into user-provided document content. If the source text contains `<script>` or other HTML, it passes through unescaped. Consider HTML-escaping the text before applying `<mark>` tags.

## Router permission gap

In `universal-search.router.ts`, the `search()` method constructs permissions:

```typescript
permissions: { userId: validated.userId, isAdmin: false },
```

The `PermissionFilter` grants admins access to all documents. Because the router hardcodes `isAdmin: false`, no API caller can ever reach the admin path. The universal search tests exercise the admin path via direct service calls, but the router test never verifies it. Given the existing `SearchRouter` follows a similar pattern, this is likely an intentional v1 simplification — but it should be documented or wired to an auth layer before production. (confirmed)

## RAG call has no error boundary

```typescript
// Step 6: If aiMode, run RAG synthesizer
if (aiMode && filteredResults.length > 0) {
  const ragContexts = filteredResults.slice(0, 20).map(/* ... */);
  ragAnswer = await this.ragSynthesizer.synthesize(query, ragContexts);
}
```

If the AI engine is unreachable, rate-limited, or times out, this await throws and the entire `search()` method rejects. Steps 1-5 have already completed successfully — the permission-filtered, snippet-highlighted results exist in memory — but they're discarded because the exception propagates before the return statement. A try/catch that sets `ragAnswer = undefined` on failure would preserve the core search functionality. (confirmed)

## Sequential scope iteration

```typescript
for (const scope of searchScopes) {
  const scopeResults = await this.pipeline.search(/* ... */);
  allResults.push(...scopeResults);
}
```

`FindSimilarService.findSimilar()` in the same change correctly uses `Promise.all` for concurrent collection search. With 3 scopes at 100ms each, this adds 200ms of unnecessary serial latency. The fix is trivial — map to promises, await all, flatten. (confirmed)

## Pagination offset not implemented

The `UniversalSearchOptions` interface accepts `page?: number` and the Zod schema validates it (default 1), but the orchestrator ignores it:

```typescript
const paginatedResults = filteredResults.slice(0, limit);
```

This always returns page 1. Callers who pass `page: 2` get the same results. Either implement `slice((page - 1) * limit, page * limit)` or remove the field. (confirmed)

## Snippet highlighting and XSS surface

The `SnippetHighlighter.applyHighlighting()` method does:

```typescript
return snippet.replace(pattern, `${openTag}$1${closeTag}`);
```

The `snippet` variable is raw document text. If that text contains `<script>alert('xss')</script>`, the output will include it verbatim, interleaved with `<mark>` tags. Any frontend that renders this snippet as `innerHTML` is vulnerable. The fix is to HTML-escape the snippet text before applying the highlight regex (escaping `<`, `>`, `&`, `"`). The test suite tests regex special characters but not HTML special characters. (confirmed)

## Content extractor PDF fallback pollution

The PDF extractor's primary path uses BT/ET operator parsing. When that yields nothing (compressed streams, CID fonts — the majority of modern PDFs), the fallback extracts any 4+ character printable ASCII run:

```typescript
const printableRegex = /[\x20-\x7E]{4,}/g;
```

The filter excludes a handful of structure tokens (`obj`, `endobj`, etc.) but passes through font names (`TimesNewRoman`), encoding tables, and base64 image fragments. These will be indexed as searchable text, producing false-positive matches. (likely)
