# Real ML Serving: Triton client, feature pipeline, hybrid search, A/B experimentation, and user agency (v2)

Phase 22 replaces the in-memory random-scoring NeuralCF with a Triton Inference Server client, builds a real-time feature aggregation pipeline over NATS, adds a hybrid search system (BM25 + vector + Cohere reranker), introduces an A/B experiment framework with sticky bucketing and guardrail auto-rollback, and ships user-agency features (time-well-spent, following-mode, show-me-less, feed explainability). The v1 review's 7 issues have all been addressed: FollowingMode now delegates pagination to the content store, FeatureAggregator prunes events older than 7 days, NatsFeatureConsumer exposes an `onError` callback, Helm values document the GPU resource trade-off, `flushToOnlineStore` has per-user error isolation, `EmbeddingService.embed()` detects language per-text, and NeuralCF uses a lazy dynamic import for the test fixture fallback.

**Watch for:** `UGCEmbeddingIndexer.indexBatch` takes the first item's language and applies it to the entire batch, misrouting mixed-language content through one backend (confirmed). ScreenTimeMonitor `startSession` silently overwrites active sessions, losing engagement-trap detection state (confirmed). The `onError` callback in NatsFeatureConsumer and the pruning threshold in FeatureAggregator are both untested (confirmed).

## High-level view

The feature pipeline's memory leak is fixed via pruning, the flush loop has per-user error isolation, and the NATS consumer exposes an `onError` callback. However, none of these three fixes have corresponding test coverage -- the pruning branch, error-isolation loop, and onError invocation are all unexercised.

`EmbeddingService.embed()` now routes per-text instead of per-batch, correctly handling mixed-language inputs. But `UGCEmbeddingIndexer.indexBatch` passes `batch[0]?.language` as the explicit language for the entire sub-batch, which overrides per-text detection and forces all items in a mixed-language batch through one backend.

ScreenTimeMonitor's `startSession` unconditionally overwrites existing sessions. In mobile apps where the method fires on every foreground transition, the engagement-trap counter resets silently -- the feature designed to detect problematic usage patterns fails precisely when the user re-opens the app mid-session.

## UGCEmbeddingIndexer batch language misrouting

In `embedding-indexer.ts`, `indexBatch` takes the first item's language and passes it as the explicit language argument:

```typescript
async indexBatch(items: IndexableContent[]): Promise<void> {
  for (let i = 0; i < items.length; i += this.batchSize) {
    const batch = items.slice(i, i + this.batchSize);
    const texts = batch.map((item) => item.text);
    const language = batch[0]?.language;

    const embeddings = await this.embeddingProvider.embed(texts, language);
    // ...
  }
}
```

When `language` is defined (e.g., `'en'` from the first item), `EmbeddingService.embed()` takes the explicit-language path and routes all texts to a single backend. A batch of 100 items where items 50-60 are Hindi but item 0 is English will embed all 100 through OpenAI. The fix is straightforward: either don't pass `language` (let per-text detection handle it), or pre-group items by language before batching.

## ScreenTimeMonitor session state loss

```typescript
startSession(userId: string): void {
  this.sessions.set(userId, {
    sessionStart: Date.now(),
    lastItemTime: 0,
    consecutiveRapid: 0,
    optedOut: false,
  });
}
```

No guard against an already-active session. In mobile apps where `startSession` fires on every foreground transition, the engagement-trap counter (`consecutiveRapid`) and accumulated session duration reset to zero. A user scrolling rapidly through content, then locking/unlocking their phone, would have their engagement-trap detection silently reset. Adding an early return when `this.sessions.has(userId)` (or merging state) would preserve detection continuity.

## Test coverage gaps for v1 fixes

The NatsFeatureConsumer test calls `handleMessage` with malformed data and asserts `processEvent` was not called, but never passes `options: { onError }` to verify the callback fires.

The aggregator test pushes a few events and verifies sliding-window logic, but never pushes >100 events with timestamps older than 7 days to trigger the pruning branch. The pruning condition `state.events.length > 100 && state.events[0]!.timestamp < cutoff` is untested.

The `flushToOnlineStore` test verifies happy-path flush and clean-state skip, but never mocks `setFeatures` to reject and verify that subsequent users still flush.
