# Real SDK integrations replacing stubs across 9 services

Phase 10 rips out the in-memory stub implementations across payments, media, notifications, social graph, search, SMTP, ad auction, dating matches, and CDN. Each is replaced with a thin wrapper around the real SDK (Stripe, fluent-ffmpeg, Firebase Admin, node-apn, MeiliSearch, smtp-server/nodemailer, ioredis, pgvector via Prisma raw SQL, AWS CloudFront). The pattern is consistent: a class wrapping the SDK, Zod validation at the boundary, and tests that mock the SDK at the module level. Backward-compatibility aliases preserve the old export names.

**Watch for:** The auction service has an asymmetric cleanup path where single-bidder auctions skip Redis key deletion (confirmed). The pgvector query interpolates a vector string literal directly into a tagged template, which may fail under Prisma's parameterization rules depending on database driver (likely). The PushService uses `notification.topic = 'com.quant.app'` as a hardcoded APNs topic, which must match the app's bundle ID in production (confirmed).

## High-level view

The media package takes on two distinct SDK integrations: fluent-ffmpeg for HLS transcoding (with a master playlist generator that writes real m3u8 content) and CloudFront for presigned URL generation and cache invalidation.

The push notification service collapses a 477-line stub into 180 lines that delegate to Firebase Admin for Android/Web and node-apn for iOS. Multicast is FCM-only; there is no APNs equivalent path for bulk sends. The web push path has a copy-paste bug assigning `imageUrl` to the `badge` field.

The social graph service uses a `PrismaWithRelationship` interface cast to avoid needing a generated Prisma client at compile time, and layers Redis caching for follower counts. The `getMutualFollowers` method pulls unbounded result sets into memory for client-side intersection.

The auction service implements a second-price (Vickrey) mechanism using Redis sorted sets. It has a correctness gap: only the two-bidder path cleans up Redis keys after resolution; the single-bidder path leaves keys behind. Neither path cleans up bid metadata hashes.

The pgvector matching service finds matches using raw SQL with the `<=>` cosine similarity operator. The vector string is interpolated into a Prisma tagged template, which parameterizes values and may break the `::vector` cast depending on driver behavior.

The SMTP inbound service is cleanly split: `SmtpInboundServer` handles the protocol and streams to mailparser, while `EmailSender` wraps nodemailer. `EmailSender.send` throws on Zod validation failure but returns `{ success: false }` on transport failure, creating an asymmetric error contract for callers.

## Auction resolution asymmetry and metadata orphans

The `resolveAuction` method has two code paths. When two or more bidders exist, it deletes the bid key and removes the auction from the active set. When only one bidder exists (array length 2), it returns the winner at their own price but skips both cleanup calls. The single-bidder resolved auction remains in the active set and its sorted-set key persists indefinitely.

```typescript
// Single-bidder path (lines ~92-104) — no cleanup
return {
  winnerId,
  winningBid: bidAmount,
  secondPrice: bidAmount,
  effectiveScore: winningScore,
};

// Two-bidder path (lines ~116-118) — cleanup present
await this.redis.del(key);
await this.redis.srem(AuctionService.ACTIVE_AUCTIONS_KEY, auctionId);
```

Neither path deletes the `auction:meta:{auctionId}:{bidderId}` hash keys. After resolution, these metadata hashes become orphans. The `cancelBid` method correctly deletes its own metadata key, showing the pattern is known but not applied to resolution.

## pgvector query parameterization risk

The matching service constructs a vector string and passes it into a Prisma tagged template:

```typescript
const vectorStr = `[${embedding.join(',')}]`;
const results = await this.prisma.$queryRaw<...>`
  SELECT ... 1 - (dp.embedding <=> ${vectorStr}::vector) as similarity
  ...
  ORDER BY dp.embedding <=> ${vectorStr}::vector
  LIMIT ${limit}
`;
```

Prisma's `$queryRaw` with tagged templates parameterizes interpolated values. The `${vectorStr}::vector` syntax expects the driver to pass the string as a parameter and apply the `::vector` cast server-side. Whether this works depends on whether the Postgres driver can successfully cast a parameterized string value to the `vector` type. If it fails, the query will throw at runtime. This should be validated with an integration test against a real pgvector-enabled database, or the vector string should be embedded via `Prisma.sql` / `Prisma.raw()` to bypass parameterization.

## Push notification WebPush badge bug and multicast gap

The `sendFcm` private method has a bug in its web push path:

```typescript
message.webpush = {
  notification: {
    title: payload.title,
    body: payload.body,
    badge: payload.imageUrl,  // <-- imageUrl assigned to badge
  },
};
```

The `badge` property in WebPush notifications expects a URL to a badge icon (a small monochrome image), not the notification's image URL. This is a copy-paste error.

The `sendMulticast` method only goes through FCM. The API accepts a flat array of tokens with no platform discrimination. If a caller passes iOS device tokens into multicast, they'll be sent to FCM (where they're meaningless) and return per-token failure results.

## Social graph unbounded mutual followers query

The `getMutualFollowers` method fetches all followers of both users into memory, then computes the intersection client-side. For users with large follower counts, this pulls unbounded result sets. A single SQL query with `INTERSECT` or a join would be more efficient and bounded.

The `PrismaWithRelationship` interface and double-cast through `unknown` at every call site suppresses type safety between the real Prisma client and this service. Every Prisma query shape change requires updating both the interface and all cast sites.

## SMTP sender error contract asymmetry

`EmailSender.send` validates options via Zod before calling `sendMail`. Invalid addresses throw synchronously (Zod error). Valid addresses that fail at the SMTP transport level return `{ success: false, error: ... }`. Callers need to handle both thrown exceptions and error results. The `sendPush` method in the notification service handles this same pattern gracefully by wrapping the entire flow in try/catch; `EmailSender.send` does not.

## Test coverage gaps

- **Not tested:** Concurrent bid submission to the same auction (race conditions in `submitBid`).
- **Not tested:** The `writeMasterPlaylist` method's m3u8 content format correctness.
- **Not tested:** `EmailSender` behavior when constructed without explicit config (env var fallback path).
- **Not tested:** `SocialGraphService.block` removing existing FOLLOW relationships (only implicitly covered through mock setup).
