# Wave 1 wiring: live chat, SSE streaming, mail proxy, token management

This change converts three stub apps into working frontends by adding Next.js API route proxies, wiring WebSocket broadcasts alongside REST persistence in QuantChat, rewriting the QuantAI chat route from JSON-only to a streaming SSE proxy, completing QuantMail's proxy layer with inline reply and AI summarize, and introducing a shared TokenManager + HttpClient auto-refresh mechanism in the api-client package.

**Watch for:** The QuantChat token refresh consumes the 401 response body before checking status, so the retry sends the stale (failed) parsed data back to the caller instead of the refreshed response (confirmed). The SSE proxy routes mask all backend errors behind HTTP 200, making client-side error handling and monitoring dependent on parsing SSE data payloads (confirmed). The QuantMail thread reply bypasses the api-client entirely, using raw `fetch` with no auth header (confirmed).

## High-level view

The SSE streaming proxy in QuantAI is the structural centerpiece. The old route parsed JSON and returned it; the new version detects `stream: true`, opens a streaming fetch to the backend, and pipes the ReadableStream through to the client. Both routes propagate client disconnects to the backend via AbortController. However, all error cases — backend 4xx/5xx, network failures, missing body — are wrapped in HTTP 200 SSE payloads, making failures invisible to HTTP-level monitoring.

QuantChat's real-time messaging wires dual-write: every message POST goes to the REST API for persistence and simultaneously broadcasts over the existing WebSocket layer. The token refresh in the app-level api-client has a fail-open bug where a failed refresh still returns the parsed 401 body to callers.

QuantMail completes its proxy surface with forward, batch, thread reply, and AI summarize routes. The inline reply form and summarize button bypass the typed `apiClient` and use raw `fetch` without auth headers — an asymmetry with the rest of the page's operations which all go through `apiClient`.

The shared TokenManager hydrates from localStorage and wires into HttpClient via `createApiClient`. HttpClient deduplicates concurrent 401 refresh attempts via a shared promise, preventing thundering-herd refreshes.

<details>
<summary>Issues (6)</summary>

1. **QuantChat 401 refresh uses stale body** — `request()` parses `response.json()` unconditionally, then checks for 401 and retries. On refresh _failure_, the already-consumed 401 error body is returned as if it were a valid response. Check status before consuming body, or return an explicit error after `onAuthError` fires.
2. **SSE routes always return 200 on backend error** — Both `/api/assistant/chat` and `/api/ai/stream` wrap backend 4xx/5xx in an SSE payload with status 200. Monitoring, error-rate dashboards, and HTTP error interceptors will never see these failures. Consider returning the actual status code for non-streaming errors, or emit a structured `event: error` SSE event type.
3. **QuantMail inline reply skips auth** — `handleSendReply` uses raw `fetch` with no Authorization header while other operations on the same page use `apiClient`. Route through `apiClient` or forward the token.
4. **QuantMail summarize ignores errors** — `handleSummarize` doesn't check `res.ok`. A 500 from the backend results in silent failure with no user feedback.
5. **Proxy helpers crash on non-JSON responses** — Both `proxyToBackend` implementations call `res.json()` unconditionally. If the backend returns HTML, empty 204, or a gateway error, this throws an unhandled exception. Add try/catch with a fallback error response.
6. **No test coverage for new code paths** — None of the new files (proxy helpers, SSE routes, TokenManager, useModels, inline reply) have tests. The ~1100 new lines are covered only by pre-existing tests that don't exercise them.

## QuantChat token refresh consumes body before status check

In `apps/quantchat/src/services/api-client.ts`, the `request()` method does:

```typescript
const data = (await response.json()) as ApiResponse<T>;

if (response.status === 401 && this.refreshToken) {
  const refreshed = await this.refreshTokens();
  if (refreshed) {
    headers['Authorization'] = `Bearer ${this.accessToken}`;
    const retryResponse = await fetch(url, { ... });
    return retryResponse.json() as Promise<ApiResponse<T>>;
  }
  this.onAuthError?.();
}

return data;
```

If `refreshTokens()` returns false, `onAuthError?.()` fires and then the method falls through to `return data` — returning the 401 error body as if it were a valid response. The caller has no way to distinguish this from a successful response unless it inspects `data.success`, which the callers (hooks) may not consistently do.

The QuantMail client has the same structural issue. The shared `HttpClient` in `packages/api-client` does this correctly by checking `!response.ok` before consuming the body.

## SSE error masking behind HTTP 200

Both streaming routes return HTTP 200 for every case — including when the backend returns 4xx/5xx or throws:

```typescript
controller.enqueue(encoder.encode(`data: ${JSON.stringify({ error: errorText })}\n\n`));
controller.enqueue(encoder.encode('data: [DONE]\n\n'));
```

The client's `processSSEStream` must detect `{ error: ... }` in the data payload. Any HTTP-level monitoring (edge CDN, load balancer error rates, observability) will see 100% success rates even when the backend is down. The no-body case returns 200 with just `data: [DONE]\n\n`, which is indistinguishable from a successful empty response.

## QuantMail inline reply and summarize bypass api-client

In `apps/quantmail/src/app/thread/[id]/page.tsx`, `handleSendReply` uses:

```typescript
await fetch(`/api/threads/${threadId}/reply`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ body: replyText }),
});
```

No Authorization header. The `apiClient` instance is imported and used for archive, star, and delete on the same page, but reply and summarize bypass it. If the proxy doesn't inject auth from another source, these will 401 for authenticated users.

The summarize handler doesn't check `res.ok`, so a 500 from the backend results in silent failure — the user gets no feedback that summarization failed.

## Proxy helpers assume JSON responses

Both `proxyToBackend` implementations do:

```typescript
const res = await fetch(url.toString(), fetchOptions);
const data = await res.json();
return NextResponse.json(data, { status: res.status });
```

If the backend returns HTML (nginx error page), an empty 204, or a malformed response, `res.json()` throws. The error propagates as an unhandled exception in the Next.js API route, resulting in a generic 500 to the client with no context.

## Test gap

The diff adds ~1100 lines across proxy routes, streaming logic, TokenManager, createApiClient factory, useModels hook, and UI features. None have test files. The new behavioral paths — SSE piping, token refresh deduplication, concurrent refresh race, abort propagation, proxy error handling — are untested.
