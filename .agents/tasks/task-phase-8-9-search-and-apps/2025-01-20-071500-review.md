# Universal Search and App-Specific Product Completion (v2)

The v1 review identified 9 issues. All 9 received fixes in commit 05f497e. Seven fixes are substantive and correct: reindex API now has admin auth, orchestration routes check `DEMO_MODE`, observability has a 10,000-record eviction cap, `copyFile` re-encrypts with a fresh key, encrypted messages are excluded from search, saved search uses word-boundary matching, and `getBookmarks` uses a JSON path query. Two fixes are cosmetic only: the thread mute/snooze and bookmarks JSON queries still target schema fields that don't exist, so they will fail at the database layer regardless of the TypeScript-level fix.

**Watch for:** The admin auth hook is registered at the Fastify instance level, meaning it also gates `/healthz` and `/readyz` — Kubernetes probes will get 401 when `ADMIN_API_KEY` is set (confirmed). Thread mute/snooze writes to a `metadata` field that doesn't exist on `EmailThread` in the Prisma schema — runtime failure (confirmed). `getBookmarks` and `PostService.bookmark` reference a `metadata` field that doesn't exist on the `Post` model — runtime failure (confirmed).

## High-level view

The reindex API auth fix correctly adds an `X-Admin-Key` header check, but registers the hook at the wrong scope. Because `registerReindexRoutes` calls `app.addHook('onRequest', adminAuthHook)` on the health server's Fastify instance (not a scoped plugin), the hook fires for every route on that server, including the health and readiness probes. Any deployment with `ADMIN_API_KEY` set will break Kubernetes liveness/readiness checks.

The orchestration permission fix cleanly separates demo mode from production mode. In production, users without explicit permission entries get a 403. In demo mode, the wildcard fallback is gated behind `DEMO_MODE=true`. The empty-string userId fallback for unauthenticated production requests lands correctly on the 403 path.

The thread mute/snooze fix adds documentation and restructures the update call to write `{ metadata: { isMuted: true } }` instead of `{ isMuted: true }`. The intent is correct — store state in a JSON metadata field — but `EmailThread` has no `metadata` column in the Prisma schema. Prisma will reject the unknown field at runtime. The same pattern affects `PostService.bookmark` and `FeedService.getBookmarks`, which reference `metadata` on the `Post` model where no such field exists.

1. **Admin auth hook breaks health probes** — `registerReindexRoutes` adds `adminAuthHook` at the Fastify instance level, gating `/healthz` and `/readyz` behind `X-Admin-Key` when the env var is set. ## Admin auth hook scope on the health server

The `registerReindexRoutes` function registers the auth hook like this:

```typescript
export function registerReindexRoutes(app: FastifyInstance, jobManager: ReindexJobManager): void {
  app.addHook('onRequest', adminAuthHook);
  // ... routes
}
```

In Fastify, `addHook` at the instance level applies to all routes registered on that instance — past and future. Since the health server registers `/healthz` and `/readyz` before `registerReindexRoutes` is called, and both share the same Fastify instance, the auth hook fires on health probe requests.

The `adminAuthHook` allows access when `ADMIN_API_KEY` is not set (development mode with a warning log). In production with the env var set, probes receive 401 unless they include `X-Admin-Key`. The fix is to use Fastify's encapsulation model: register the reindex routes inside `app.register((scope) => { scope.addHook(...); scope.post(...); }, { prefix: '/api/v1/reindex' })` so the hook is scoped to the plugin.

## Schema gaps in thread and bookmark persistence

The `EmailThread` model in `packages/database/prisma/schema.prisma` has these fields:

```
id, userId, subject, participantAddresses, messageCount, isRead, isStarred,
lastEmailAt, createdAt, updatedAt
```

No `metadata` field. The `muteThread` and `snoozeThread` methods attempt to write `{ metadata: { ... } } as never` — the `as never` cast silences the TypeScript compiler, and the unit tests pass because they mock Prisma. At runtime against a real database, Prisma's generated client will reject the unknown field in the `data` object with a validation error (Prisma Client validates the shape before sending SQL in Prisma 4+).

The same pattern exists for `Post`:

```
id, userId, type, content, mediaUrls, hashtags, mentions, linkPreview, replyToId,
communityId, visibility, likeCount, commentCount, repostCount, viewCount, isEdited,
isPinned, moderationStatus, publishedAt, createdAt, updatedAt, deletedAt
```

No `metadata` field. `PostService.bookmark` writes `{ metadata: { ...metadata, bookmarkedBy } }` and `FeedService.getBookmarks` uses `metadata: { path: ['bookmarkedBy'], array_contains: [userId] }`. Both will fail at runtime.

The unit tests mask this because they mock `prisma.post.update` and `prisma.post.findMany` — the mocked functions accept any arguments. The TypeScript compiler is silenced by the `as never` cast or by the untyped `where` object construction. A schema migration adding `metadata Json @default("{}")` to both models is the minimal fix.
