# React Query wire-up across 7 frontend apps (v2)

The fix commit addresses 6 of the 7 issues from the v1 review. `useFeed`, `useMatching`, `useLive`, and `useCreator` in QuantMax are now wired to the API client via React Query. The dead `useForYouFeed` hook is deleted. The `shows.tsx` page is fully rewritten with a new `useShows` hook. The matching page correctly calls `matching.like(profileId)` instead of the non-existent `matching.swipe('like')`. The `as any` casts in quantchat, quantmail, and quantneon/useFeed were removed.

Watch for:

- **`useFeed` pagination is non-functional** (confirmed) — `useInfiniteQuery` increments `pageParam` but never passes it to `apiClient.getForYouFeed(20)`, so every "page" fetches the same first 20 videos.
- **Render-time side effect in `useFeed`** (confirmed) — `feedQuery.fetchNextPage()` is called synchronously during render rather than inside a `useEffect`, which can trigger infinite re-render loops in React strict mode.
- **`useLive` fetches streams but discards the data** (confirmed) — `liveStreamsQuery.data` is never consumed; `viewers`, `recentGifts`, and `topGifters` are always returned as empty arrays.

**Verdict**: NEEDS_CHANGES

## High-level view

The seven v1 issues were systematically addressed. QuantMax hooks that were pure local-state simulations now import `@tanstack/react-query` and call through `apiClient`. The matching page's runtime-error-inducing API mismatch is fixed: it calls `like(id)`, `pass(id)`, `superLike(id)` directly. The QuantTube `shows.tsx` was properly rewritten with a `useShows` / `useShow` pair that integrates genre filtering and episode detail through the API.

Three new issues emerged from the fix. The `useFeed` hook adopted `useInfiniteQuery` but the underlying API method doesn't accept pagination parameters, so `fetchNextPage` produces duplicate data. That same hook calls `fetchNextPage` during render rather than in an effect. The `useLive` hook queries `['live-streams']` to drive its `isLoading` flag but never exposes the fetched stream list to the UI, leaving `viewers`, `recentGifts`, and `topGifters` permanently empty.

## `useFeed` infinite query without pagination parameters

The `useInfiniteQuery` in `useFeed.ts` increments a `pageParam` counter and uses it to derive `getNextPageParam`, but the actual fetch function ignores pagination entirely:

```ts
queryFn: async ({ pageParam = 0 }) => {
  const response = await apiClient.getForYouFeed(20); // pageParam not passed
  return { videos: response.data ?? [], nextPage: pageParam + 1 };
},
```

`apiClient.getForYouFeed` only accepts `limit`. Every call returns the same 20 videos. The `getNextPageParam` logic compares `lastPage.videos.length < 20` to decide whether more pages exist, but since the same results come back each time, it either loops infinitely or (if the API returns fewer than 20) stops after one page. Either way, infinite scroll doesn't produce new content.

The auto-load trigger compounds this: it runs as a bare conditional in the component body (`if (currentIndex >= allVideos.length - 3 && ...) feedQuery.fetchNextPage()`). Under React 18 strict mode, the component renders twice. If `fetchNextPage` triggers a state update on `feedQuery`, the component re-renders, the condition is re-evaluated, and another `fetchNextPage` fires. This should be a `useEffect` with proper dependencies.

## `useLive` data flow gap

The hook queries `apiClient.getLiveStreams()` and surfaces `liveStreamsQuery.isLoading` to consumers. But the actual stream list is discarded:

```ts
return {
  stream,          // local state, only set when the user goes live
  viewers: [],     // hardcoded
  recentGifts: [], // hardcoded
  topGifters: [],  // hardcoded
  isLoading: liveStreamsQuery.isLoading,
  ...
};
```

The `live.tsx` page destructures `topGifters` and renders a leaderboard, but it will always be empty. The `viewers` count shown in the stream header is derived from the empty array. This means the live page appears functional but renders no real viewer data even if the API returns it.
