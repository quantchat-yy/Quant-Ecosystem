# QuantLive conversational UI component suite

A six-component UI kit for a conversational AI overlay has been added to `packages/shared-ui`. The components implement an idle/active/minimized state machine driven by the top-level `<QuantLive />` wrapper. Subcomponents handle the pulsing orb, live captions, action-in-progress chip, media controls, and an always-visible privacy indicator. The suite ships with 37 tests and clean TypeScript compilation.

**Watch for:** The privacy indicator renders as a DOM-only assertion (fixed + z-[9999]) but nothing prevents a consuming app from unmounting it or overriding its styles (**likely**). The internal state (`captions`, `currentAction`, `orbColor`) is currently unreachable from outside the component, meaning the entire active view is a non-functional shell until an integration API is added (**confirmed**). The `void setX` pattern to suppress unused-variable warnings signals significant deferred wiring (**confirmed**).

## High-level view

The state machine in `QuantLive.tsx` governs which subcomponents render via three branches (idle/active/minimized). Transitions are callback-driven (`handleActivate`, `handleDeactivate`, `handleMinimize`, `handleMaximize`) with CSS `transition-all` for the visual spring. The machine is sound for these three states, but the component exposes no mechanism for external actors (a session engine, LLM stream) to push captions, action chips, or orb-color changes into it.

Privacy enforcement relies on DOM-level positioning (`fixed top-2 right-2 z-[9999]`) and conditional rendering. The indicator hides itself entirely when all three sources are inactive, which is correct. However "cannot be hidden by app code" is an aspirational design goal, not an enforced guarantee; any ancestor can `display:none` or `overflow:hidden` the element.

Test coverage exercises the state machine transitions, ARIA labels, and positioning, but `QuantLiveCaptions`, `QuantLiveActionChip`, and `QuantLiveControls` have no dedicated test files -- the components with the most conditional rendering logic are untested.

<details>
<summary>Issues (5)</summary>

1. **No external state injection API** -- `captions`, `currentAction`, and `orbColor` are internal state with no props or ref-forwarding to set them. Consumers cannot drive the active view. Add controlled props or a ref-based imperative API before integration.
2. **Privacy indicator not structurally protected** -- The "always visible" guarantee depends on DOM z-index and fixed positioning. A parent `overflow:hidden`, `display:none`, or portal-less mounting can defeat it. Consider rendering it via a React portal attached to `document.body`.
3. **Three subcomponents untested** -- `QuantLiveCaptions`, `QuantLiveActionChip`, and `QuantLiveControls` have no dedicated test files. Caption auto-scroll, the action-chip null-renders-nothing path, and individual control toggles are uncovered.
4. **Camera toggle emoji unchanged between states** -- `QuantLiveControls` renders the same emoji (`\u{1F4F7}`) for both camera-on and camera-off, making the states visually indistinguishable beyond background color.
5. **`positionStyles` key lookup has no fallback** -- `positionStyles[position]` would return `undefined` for any position value not in the record, producing `undefined` in the className string. TypeScript catches this at compile time via the `QuantLivePosition` union, but runtime consumers passing arbitrary strings through type assertions would get a broken class.

## State machine and integration surface

The `QuantLive` component holds seven pieces of state internally: `state`, `orbColor`, `micMuted`, `cameraActive`, `screenSharing`, `captions`, and `currentAction`. Of these, only `state` and the three media booleans are actually mutated during normal operation. The remaining three (`orbColor`, `captions`, `currentAction`) are initialized and then left untouched, with their setters explicitly voided:

```tsx
void setOrbColor;
void setCaptions;
void setCurrentAction;
```

This means the active view always renders an empty captions list, a null action chip, and the `listening` orb color (set once on activation). The current API shape provides no controlled-component path (`captions` prop, `orbColor` prop) or imperative handle (`useImperativeHandle`). When integration arrives, the component will need a breaking API change or a significant refactor to accept external data.

## Privacy indicator enforcement gap

The spec states the privacy indicator "cannot be hidden by app code." The implementation uses `fixed top-2 right-2 z-[9999]` which achieves always-on-top under normal rendering. Since the indicator is rendered inline within `QuantLive`'s subtree (not portaled), any ancestor applying `display:none`, `visibility:hidden`, or being removed from the DOM will take the indicator with it. A React portal mounting directly on `document.body` would make the guarantee structurally robust rather than relying on cooperative parents.

## Camera toggle visual ambiguity

In `QuantLiveControls.tsx`, the camera button renders `\u{1F4F7}` (camera emoji) for both the active and inactive states:

```tsx
<span aria-hidden="true">{cameraActive ? '\u{1F4F7}' : '\u{1F4F7}'}</span>
```

The background color does change (`bg-gray-100` vs `bg-red-100`), and the aria-label distinguishes the states for screen readers. But the visual icon is identical, which is inconsistent with the mic button that uses distinct glyphs (`\u{1F507}` muted vs `\u{1F3A4}` active).

## Test coverage gaps

Not tested:

- `QuantLiveCaptions` -- auto-scroll behavior, opacity gradient calculation, user-vs-assistant styling, `maxVisible` truncation, `isFinal` italicization.
- `QuantLiveActionChip` -- null-action rendering, dot animation presence, icon rendering.
- `QuantLiveControls` -- individual toggle callbacks, conditional rendering of optional buttons (`onToggleCamera`/`onToggleScreen` omitted vs provided), disabled-like states.

The absence of caption and control tests is notable because those components contain the most conditional rendering logic after the main wrapper.
