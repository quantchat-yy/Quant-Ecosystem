# Stub service removal and Razorpay SDK upgrade

This change prunes 10 placeholder service directories (ads-api, ai-api, chat-api, edits-api, identity, mail-api, max-api, neon-api, sync-api, tube-api) that served no purpose beyond health-check stubs, then propagates the deletions through CI/CD workflows, Helm values, and the ingress template. Separately, it upgrades `@quant/payments` to use the real Razorpay npm SDK with HMAC-SHA256 verification when credentials are present, falling back to in-memory simulation otherwise. A phase log backfill for Phases 18-40 is included as documentation.

**Watch for:** UPI live-mode verification auto-completes payments without actually polling Razorpay order status (confirmed); `createPayout` is still fully simulated even in live mode (confirmed); ingress now routes only ws-gateway traffic, meaning all HTTP API traffic for surviving services has no ingress path (likely).

## High-level view

The infrastructure cleanup is straightforward: 10 directories deleted, CI matrix reduced from 11 services to 1 (ws-gateway), and Helm values/ingress stripped of references to services that no longer exist. The only service that retains a Dockerfile is ws-gateway, so the docker-build matrices in all three workflow files now list only that service.

The Razorpay upgrade adds a dual-mode pattern where `RazorpayGateway` and `UPIPaymentService` each detect credentials at construction time and switch between real SDK calls and the existing in-memory simulation. The HMAC-SHA256 verification path is correctly implemented for `RazorpayGateway.verifyPayment`, but `UPIPaymentService.verifyPayment` in live mode simply marks pending payments as completed without confirming status with the Razorpay API, creating a path where unverified payments could be accepted.

The ingress template now only routes `/api/ws` and `ws.*` to ws-gateway. If any surviving service needs HTTP routing, that path doesn't exist in the Helm chart anymore. This may be intentional if all traffic funnels through the WebSocket gateway, but is worth confirming.

## UPI service live-mode verification gap

`UPIPaymentService.generatePaymentLink` in live mode creates a Razorpay order via the SDK, then constructs a UPI deep link using the merchant VPA and transaction metadata. The `transactionRef` stored on the `UPIPayment` object is switched to `sdkOrder.id` (the Razorpay order ID) in live mode, which is correct for later lookup.

The problem is in `verifyPayment`:

```typescript
if (this.liveMode && this.keySecret) {
  if (payment.status === 'pending') {
    payment.status = 'completed';
  }
  return { verified: payment.status === 'completed', payment };
}
```

This marks any pending payment as completed without checking Razorpay's order status. In production, a payment link could be generated but never paid by the user, and a call to `verifyPayment` would still return `verified: true`. The service does expose a separate `verifyPaymentSignature` method that does proper HMAC verification, but it's not wired into the main `verifyPayment` flow. This is a fail-open payment verification path.

## Payout simulation in live mode

`RazorpayGateway.createPayout` has a comment acknowledging that "Payouts require RazorpayX which has different endpoints" and then unconditionally simulates. In live mode, the service will appear to succeed but no money moves. The `setTimeout(() => { payout.status = 'completed' }, 0)` fires on next tick regardless of mode, so downstream consumers would see a completed payout that never happened. Any code path relying on payout status (e.g., creator cashout) would believe funds were disbursed.

## Ingress topology after cleanup

The ingress previously routed 10 path prefixes (`/api/identity`, `/api/chat`, `/api/mail`, etc.) to their respective services. After cleanup, only `/api/ws` and the `ws.*` subdomain remain, both pointing to ws-gateway on port 8080.

If Razorpay webhooks need to POST to a callback URL (which is standard for payment verification in live mode), there's no ingress rule that would route that traffic to any HTTP handler. This might be intentional if the ws-gateway itself handles HTTP routing internally, but given that the payments upgrade introduces live Razorpay integration that relies on webhooks, the gap is worth confirming.

## Test coverage for new UPI behavior

The UPI service gained three new behaviors in this PR: Razorpay order creation with `method: 'upi'`, UPI deep link construction with merchant VPA, and a `verifyPaymentSignature` HMAC method. None of these are covered by tests. The existing `upi-payment.service.test.ts` was not modified and only exercises the no-config (fallback) constructor path. By contrast, the Razorpay gateway received dedicated live-mode tests with a mock client, showing the pattern is known but wasn't applied to UPI.
